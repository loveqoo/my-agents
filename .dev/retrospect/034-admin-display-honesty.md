# 034 — 어드민 표시 정직화(승인 배지·A2A liveness) 회고

> 스펙 045(마스터 044 배치 1). UI 테스트 #12(배지 2 vs 목록 8)·#2(A2A 무조건 online).
> 테마: **어드민이 보여주는 수·상태가 실제와 일치**하게.

## 무엇을 했나

- **#12**: 사이드바 승인 배지가 mock 상수(`PENDING_APPROVALS=2`)였고 목록은 백엔드 전량(8)이라
  항구적 거짓 → 배지를 `GET /approvals?status=pending` 실수치로, 목록을 pending-only로. 백엔드
  `list_approvals(status=None)` 선택 필터(기본 전량=회귀 가드), 프론트 `listApprovals(status?)`,
  AdminShell `pendingCount` 상태 + ApprovalsView `onPendingChange` 단일 소스.
- **#2**: `register_external_agent`가 카드 검증은 하면서 status는 무조건 online → `probe_endpoint`
  (SSRF 가드 후 짧은 HTTP 도달성 probe) 추가, 도달=online·미도달=offline(등록은 안 막음).
- **사용자 추측(#2 "테스트 없이 추가") 반증 보고**: 카드는 *이미* 검증됨(SSRF 가드 포함). 진짜
  공백은 *엔드포인트 liveness*와 *가짜 시드*였다. memory probe-deeper대로 단정 전에 한 겹 더 파서
  사용자 추측을 그대로 코딩하지 않고 실체를 짚어 보고.

## 배운 것 (핵심)

**1. "정직성 결함"은 두 종류이고 고치는 층이 다르다 — 코드 거짓 vs 데이터 거짓.**
#12는 사실 두 버그가 겹친 것: (a) 배지가 mock 상수를 읽는 *코드* 거짓, (b) 시드된 가짜 승인 행이
DB에 *데이터*로 남은 거짓. 045는 (a)만, 050(파괴적 정리 dry-run)이 (b)를. 한 증상을 한 배치에
욱여넣지 않고 *고치는 층*으로 갈랐더니, 045는 순수 코드 diff(리뷰·롤백 쉬움)로, 050은 데이터
삭제(dry-run 필요)로 깔끔히 분리됐다. → learning 043.

**2. 적대 리뷰가 또 "내가 상상 못 한 범주"를 잡았다 — 이번엔 보안 회귀(리다이렉트 SSRF).**
probe에 SSRF 가드(`guard_url`)를 *깔았다는 사실*이 오히려 안심시켜 `follow_redirects=True`를 못
봤다. 가드는 *최초 URL*만 검사하므로 공개 호스트가 302로 내부 IP를 가리키면 그대로 우회된다. 내
단위 테스트는 httpx를 목으로 막아 *구조적으로* 이 구멍을 볼 수 없었다(목은 리다이렉트를 안 한다).
learning 038("적대 리뷰는 상상 못 한 범주를 찾는다")의 재확인 + 더 날카로운 따름정리: **"안전장치를
깔았다"가 "안전하다"의 증거가 아니다.** 장치의 *적용 범위 경계*(최초 URL만 vs 추종 체인 전체)를
적대자가 짚는다. → learning 044.

**3. 리뷰의 Low 지적을 그대로 "고치기"가 더 나쁜 버그를 낳을 뻔했다 — 동시성.**
"side-effect를 setQueue 밖으로"라는 Low 지적을 글자대로 받아 `queue.filter(...)`를 클로저로 꺼냈더니,
서로 다른 카드를 연속 resolve할 때 stale 클로저가 제거된 항목을 *되살리는* 회귀가 생겼다. 함수형
updater는 동시성 정답이라 버리면 안 됐다. 정답은 "updater 유지 + 외부 콜백만 밖에서 1회". → 리뷰
지적은 *방향*이지 *패치*가 아니다. 수정안의 동시성·재진입을 내가 한 번 더 검증해야 한다.
