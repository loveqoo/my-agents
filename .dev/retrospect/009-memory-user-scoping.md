# 009 — 메모리 유저 스코핑 회고 (세션 단기 ↔ 유저 장기)

날짜: 2026-06-24
브랜치: `feat/agent-service` (main 머지·push 금지 — 사용자가 직접 테스트)
지배 스펙: [018](../../docs/spec/018-memory-user-scoping.md)
이전 회고: [008-mobile-responsive-admin](./008-mobile-responsive-admin.md)

## 루프 개요
"메모리 연동" 작업. 범위가 모호한 채 시작 → Context/Planning에서 **질문으로 의도를 좁혀**
데이터 모델을 확정한 뒤 실행. mem0 `user_id` 축에 스코프를 실어 **세션 단기 ↔ 유저 장기**를
키 하나로 가르는 설계. 5단계에서 서브에이전트 비판 리뷰로 P2/P3를 잡아 반영.

핵심 결정 흐름:
1. 사용자가 "유저 식별은 선택, 없으면 세션 단기 / 있으면 mem0 장기" 모델 제시.
2. 유저 추출 방식 = 명시적 `userId` 필드(결정 A). 인증주체(B)·mem0 자동(C) 기각.
3. 사용자 두 미지수 제기(A2A 유저 유입 / checkpoint thread_id 연결) → **추측 금지** 준수,
   코드 현황 + 웹 문서로 확인: 둘 다 미구현(플래그/목업), thread_id와 user_id는 **직교축**.
4. "유저 식별 토글 폐기, userId 필드 유무로 결정"으로 단순화(사용자 번복 수용).

## 무엇이 잘됐나
- **추측 대신 1차 출처 확인(표준 제약 준수)**: A2A·checkpoint를 "있겠거니" 넘기지 않고
  `grep`으로 코드 현황 확정(둘 다 플래그/목업) + 웹으로 thread_id↔user_id 직교성 확인.
  덕분에 "지금 A2A/checkpointer를 안 만들어도 미래와 정합"임을 근거로 범위를 좁혔다.
- **타자 검증이 진짜 버그를 잡음**: 서브에이전트가 mem0 2.0.7의 `user_id` **내부 공백 거부**를
  지적 → reviewer 주장을 믿지 않고 **설치된 mem0 소스(main.py:139-165)를 직접 확인**해 사실 검증
  → `field_validator`로 경계 정규화. 자가검증이었으면 "userId 그냥 통과"로 놓쳤을 함정(트레이스는
  `user:john doe`인데 실제론 조용히 무력화 = 데이터 유실).
- **한 값으로 키=라벨 통합**: P2(스코프/라벨 truthiness 불일치 취약성)와 P3(keyspace 충돌)을
  `mem_scope = "user:.." | "session:.."` 단일 접두사 값으로 동시 해소. 증상별 땜질 대신 표현 통합.
- **의도 좁히기 질문이 효과적**: 모호한 "메모리 연동"을 갈림길 3개(누구의 기억/유저 정의/가시화 vs 모델)
  로 제시 → 사용자가 데이터 모델을 명료화. Planning 규칙("의도 잡힐 때까지 질문") 준수.

## 무엇이 잘못됐나 / 배운 것
- **첫 설계에 토글을 넣었다가 번복**: "유저 식별은 선택"을 *에이전트별 토글*로 해석했으나 사용자는
  *요청 필드 유무*를 의도. 한 번 더 확정 질문(1 vs 2)으로 막판에 정렬했지만, 처음 "선택"의
  주체(에이전트 설정 vs 호출)를 더 일찍 못 박았으면 스펙 재편집을 줄였을 것.
- **스펙 초안이 실행 중 바뀜**: prefix·validation은 타자 검증 후에야 스펙에 반영. 검증에서 나온
  설계 변경을 스펙에 역반영하는 건 정상이지만, mem0 제약(공백 거부)은 **Context 단계에서 mem0
  소스를 미리 봤으면** 초안에 넣을 수 있었다 — 외부 라이브러리 제약은 설계 전에 확인하는 게 낫다.
- **라이브 통합 검증은 미완**: 세션격리/유저장기 회상은 MLX+qdrant 스택 필요 → 사용자 브랜치 테스트로
  이관. 정적/로직/단위는 닫았으나 end-to-end는 사용자 손에. (표준 제약상 맞는 분담)

## 다음에 다르게 / 추후
- **외부 라이브러리 키/식별자 제약은 Context에서 선확인**: mem0처럼 entity-id 규칙(공백/빈값 거부)이
  있는 SDK는 설계 전에 소스/문서 확인. [[019-mem0-memory-scoping]]에 정리.
- **"선택"류 요구는 주체를 즉시 확정**: 무엇을(축) + 어디서(에이전트 설정 vs 요청 vs 인증)를 한 번에.
- 미완 추후: 실 A2A 핸들러의 신원 추출(토큰/metadata→userId), LangGraph 체크포인터(thread_id=session),
  세션 종료 시 단기 메모리 purge, 장기·일화적/절차적 타입 배선.
