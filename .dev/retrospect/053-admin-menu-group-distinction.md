# 053 — 어드민 메뉴 그룹 구분 회고

스펙 065. 사용자 제안("어드민 기능과 아닌 것은 메뉴에서 구분")을 받아 슈퍼유저 전용 메뉴
항목(유저·배치·허용 호스트)을 `type:'group'` "관리자" 헤더로 묶어 일반 작업 메뉴와 시각 분리.

## 무엇을 했나
- AdminShell.tsx 슈퍼유저 분기를 평면 항목 배열 → 단일 그룹 항목으로 교체("도구" 그룹과 동일
  패턴, collapsed 시 라벨 숨김). 키·views·게이트 무변경.
- 브라우저 3샷(확장/collapsed/그룹-항목-네비)으로 검증, tsc 무에러.

## 핵심 통찰
1. **"노출 게이트(보안/권한)"와 "시각 구분(표현)"은 별개 축이다.** 슈퍼유저 항목은 이미
   `user.is_superuser` 게이트로 *노출*은 분리돼 있었고(서버는 064 §D5 `require(...,"manage")`로
   독립 강제), 빠진 건 *표현*뿐이었다. 요청을 받자마자 "권한 로직"으로 끌고 가지 않고 "메뉴에서
   구분"이 순수 presentational임을 먼저 분별 → 인증/배선을 건드리지 않고 메뉴 배열만 재배치.
   요청의 층(보안 vs UX)을 먼저 가르면 변경 표면이 최소가 된다.
2. **in-file 선례 패턴이 있으면 그게 곧 설계다.** "도구" 그룹(`type:'group'`+collapsed 라벨 숨김)이
   이미 있어 새 antd 패턴 학습·learning full 통독 없이 1-블록 변경으로 끝. Context에서 인덱스 후크가
   "충분"하다고 판단해 full을 안 연 것이 적중(index-layer-for-context-recall 실천).

## 검증 판단
순수 표현 변경(비가역·보안 표면 없음)이라 타자 적대(codex) rung은 스킵하고 브라우저 실측+타입체크로
충분. 064(보안/파괴 경로)와 달리 "검증을 작업 성격에 맞게"의 반대편 사례 — 모든 변경에 동일 사다리를
기계적으로 적용하지 않고, 표면(보안/비가역 여부)으로 rung을 고른다.

## 별도 learning 미작성(사유)
이식 가능한 *실패 패턴*이 새로 드러나지 않음(선례 패턴 차용·무버그). 통찰 1(요청의 보안 vs 표현
층 분별)은 회고로 충분 — 강제로 learning을 만들면 노이즈. 같은 분별이 재발·실수로 이어지면 그때
learning으로 승격.
