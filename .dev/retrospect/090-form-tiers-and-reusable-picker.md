# 090 — 폼 3단계 + 재사용 효율 피커(등록 폼 + 오버라이드) (스펙 109)

## 무엇을 / 왜

108 이후에도 사용자 지적: (1) 필수/공통 vs 종류별/선택이 안 나뉘어 평평하다, (2) 늘어나는 항목을
효율 렌더해야, (3) **같은 개선을 플레이그라운드 오버라이드에도** 반영해야. 즉 UX 원칙(위계·효율
렌더)을 **두 화면에 일관되게** 적용하는 문제.

## 한 일

- **재사용 컴포넌트 `PickerGroups`**(`admin/src/PickerGroups.tsx`) 추출 — 접이식+카운트+검색+선택
  있는 그룹 기본 펼침(107 패턴을 컴포넌트로). 등록 폼·오버라이드 **둘 다** 사용 → 렌더·검색·카운트
  drift 0.
- **등록 폼 3단계**(AgentsView): 기본(이름·모델·페르소나·종류) / 이 에이전트가 하는 일(종류별
  PickerGroups) / 세부 설정(접힘: Temperature·히스토리·대화저장).
- **오버라이드 일관화**(OverridePanel): memories·mcps → PickerGroups("이 대화에서 쓸 것"),
  Temperature·history → "세부 설정" 접힘. `Overrides`·`overridePayload` 무변경(표현 계층만).

## 잘된 것

- **재사용 컴포넌트로 두 화면 일관성을 싸게** 확보(learning 109): "오버라이드에도 반영"이 컴포넌트
  하나 꽂기로 끝남. 다음 화면도 공짜.
- **어댑터로 이질적 상태를 한 컴포넌트에 통일**: 직접 자원=4개 배열, 오버라이드=2개 배열, 능력=1개
  배열 — id를 카테고리 prefix로 네임스페이스하고 toggle 어댑터로 라우팅해, **저장 구조는 그대로** 둔
  채 하나의 storage-agnostic 컴포넌트로 묶음.
- **브라우저 두 화면 다 검증**: 등록 폼 9/9(3단계·종류별 그룹·접힘·토글) + 오버라이드 6/6(피커·세부
  설정·기술 id 부재). 실화면 스샷 둘.

## 배운 것 / 함정

- **cross-surface 일관성은 재사용 컴포넌트로**(learning 109): 사용자가 "여기도 저기도 같게"를 원하면
  각 화면에 복붙하지 말고 컴포넌트를 뽑아라 — 복붙은 즉시 drift(한쪽만 고침)를 낳는다. 어댑터
  (prefix+toggle 라우팅)로 저장 구조가 달라도 한 컴포넌트로 흡수.
- **antd v6 Collapse 헤더 텍스트는 `.ant-collapse-header`**(‑text 아님) — 검증 셀렉터 함정. 체크박스
  토글은 `label.ant-checkbox-wrapper … input.check()`(107 재확인).
- **플레이그라운드 오버라이드는 ui 에이전트에서만 폼**(code/external은 read-only) — 검증 시 상단
  드롭다운으로 ui 에이전트("Research Assistant") 선택 후 열어야 함.

## 검증 안 함(사유)

- 저장/적용 로직(`overridePayload` diff, `save` config)·백엔드 무변경 — 표현 계층만 교체(회귀 뷰에 갇힘).
- 리스트 가상화는 OUT(검색+접이식으로 수백까지 충분).
