# 013 — 구조 바로잡기: Context 스킵을 막는 가시적 장치 (AI 회고)

날짜: 2026-06-25
브랜치: `feat/agent-service` (main 머지·push 금지)
유형: 절차/구조(자기 교정) — 이번 태스크 = 6단계 루프의 구조 결함 수정
참고 자산: [[013-keep-the-step-header]](동형 1차 사례), 메모리 `dont-skip-context-recall-learnings`, spec 016(시연에서 끌어올린 결정)

---

## 태스크
기능 작업이 아니라 **루프 자체의 구조 결함**을 바로잡는 것. 결함: 내가 Context(단계 2)를
상습적으로 스킵 → Compounding(회고)이 다음 작업에 상기·적용되지 못해 **복리가 0**.

## 무엇이 드러났나
- 직전(스펙 021/#9)에서 나는 시작부터 `.dev/learning/`·회고를 안 읽고 측정/재현에 뛰어들어 헛돌았다.
- 사용자 지적: "context 단계를 자주 스킵한다. 회고를 하면 뭐합니까? 다시 상기시키고 다음 작업에
  적용을 해야지요." — 정곡.
- 진단: **규칙은 이미 CLAUDE.md에 있었다**("작업 전 learning/retrospect 확인"). 그런데 *가시적
  산출물*이 없어 스킵해도 표가 안 났고, 그래서 몰입하면 그냥 건너뛰었다.

## 동형 반복 (중요)
이건 [[013-keep-the-step-header]]와 **같은 메타 패턴의 2차 사례**다: 규칙은 존재하나 가시적
강제물이 없어 몰입 중 스킵 → 사용자가 적발. 두 번 반복했다는 것 자체가 신호 → 일반 원리를
learning [[023-process-rules-need-visible-enforcement]]로 자산화.

## 처방 (사용자 승인 후 적용)
1. **CLAUDE.md 단계 2 + 빠른 합의 규칙**에 `참고 자산:` 필수 줄 규칙 추가 — grep로 관련
   learning/retrospect/spec 읽고 `[번호-제목 → 적용점]` 명시, 없으면 "관련 없음(사유)",
   줄 자체가 없으면 Context 스킵으로 간주.
2. **메모리**(feedback) `dont-skip-context-recall-learnings` + MEMORY.md 인덱스 → 차기 세션 상기.
3. **즉시 시연**: #4(메모리 수정) Context 스캔을 실제로 돌려 spec 016(메모리 타입=런타임 계약 →
   UI 읽기전용 결정)·learning 021/002·기지 제약(`search()`가 mem0 `id`를 떨굼)을 끌어올림.
   스캔 없이 했으면 016 읽기전용 결정과 충돌하는 수정 폼을 만들 뻔 → 장치의 가치 입증.

## 교훈
- **규칙 ≠ 준수.** 절차 규칙은 응답에 남는 *가시적 1줄*이 있어야 강제된다(스킵이 즉시 검출).
- 새 규칙을 만들 땐 "이걸 어떻게 가시적으로 강제하지?"를 함께 설계한다(규칙만 적고 끝내지 말 것).
- 나는 프로세스 문서를 "있으니 지켜진다"고 가정하는 경향 — 두 번째 lapse. 경계.
