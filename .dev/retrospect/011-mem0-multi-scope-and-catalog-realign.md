# 011 — mem0 다층 스코프 + 카탈로그 재정렬 회고

날짜: 2026-06-24
브랜치: `feat/agent-service` (main 머지·push 금지 — 사용자가 직접 테스트)
지배 스펙: [020](../../docs/spec/020-mem0-multi-scope-and-catalog-realign.md)
이전 회고: [010-mem0-pgvector-migration](./010-mem0-pgvector-migration.md)

## 루프 개요
"mem0가 지원하는 메모리 기능 파악"으로 시작했으나, **사용자의 연속된 이의 제기**가 작업을 두 번
재정의했다. (1) "왜 4종인가, 한 종만 쓰는가?" → 카탈로그가 mem0 기능이 아니라 인지과학 분류
데모임을 확인. (2) "여기 구현으론 mem0 홍보 수준 이점을 못 누린다, 소스 말고 웹을 봐라" →
내 '1 of 3, 이점 없음' 결론이 **틀린 모델** 위에 있었음을 웹 리서치로 교정. mem0의 실제 가치는
memory_type 스위치가 아니라 **추출 파이프라인 + 다층 스코프 + 자동 엔티티 그래프**. 그래서 작업이
"4종 카탈로그 구현"이 아니라 "단일 user_id 축을 user/session 다층으로 펼치고 카탈로그를 실제
동작과 1:1로 재정렬"로 확정됐다. agent 축은 안전한 쓰기 채널이 없어 후속으로 미뤘다.

## 무엇이 잘됐나
- **사용자가 내 결론의 토대를 흔들 때 방어하지 않고 다시 조사**: "소스만 보지 말고 웹에서 사례를
  찾아라"는 지시를 따라 mem0의 가치 모델을 교정. "당신이 옳았습니다. 제 결론은 틀린 모델 위에
  서 있었습니다"라고 명시적으로 인정하고 작업 정의를 바꿨다. 표준 제약 "추측 말고 동일 사례 확인"의
  정신을 소스 너머로 확장한 사례.
- **누출 위험을 설계로 해소**: mem0 필터가 AND라는 사실(`_build_filters_and_metadata` 소스 확인)에서
  "풍부 태깅 + 부분집합 필터 → 남의 기억 회상" 위험을 도출하고, **쓰기 축 = 사실의 소유자**(유저 턴은
  user_id+run_id만, agent_id는 안 씀) + **읽기 = 축별 단일 검색 후 합집합 병합**으로 차단. 추측이 아니라
  소스로 의미를 확정한 뒤 설계했다.
- **인프라 없이 핵심 로직을 격리 검증**: 라이브 mem0/Postgres가 필요한 부분과 순수 로직을 분리해,
  FakeMem/FakeConn으로 다축 병합·dedup·격리·마이그레이션 리맵 19종을 즉시 반복 검증(`tests/verify_020`).
  사람 의존·느린 검증을 피하라는 원칙대로.
- **타자 검증이 또 소스 대조로 잡음**: 서브에이전트가 mem0 2.0.7 `search` 시그니처가 `top_k=`(not
  `limit=`)임을 **설치 소스로 확인** → 내 `limit=limit`이 **kwargs로 삼켜져 매 축 20개 과다 fetch
  되던 결함(P3, pre-existing)을 발견. happy path(출력은 `[:limit]`로 맞음)에 가려 있던 결함.
  009·010에 이은 세 번째 "타자 검증이 happy-path 뒤를 들춘" 사례.

## 무엇이 잘못됐나 / 배운 것
- **첫 결론이 틀린 모델 위에 있었다**: "소스에 1종만 구현됨 → mem0 이점 못 누림"은 mem0를
  memory_type 분류기로 오해한 결과. 벤더가 홍보하는 라이브러리의 **가치 명제부터 외부 사례로 확인**한
  뒤 소스로 내려갔어야 했다(소스→결론이 아니라 가치명제↔소스 양방향). → 학습 [[021-mem0-value-is-scope-not-types]].
- **외부 SDK 시그니처를 또 happy-path로 통과**: `limit=`이 무시되는데도 출력 개수가 맞아 못 봤다.
  010 학습([[020-pgvector-shared-backend-and-dsn-delegation]])에 "외부 SDK에 인자 넘기기 전 시그니처를
  소스로 본다"고 적어놓고 또 키워드 인자를 확인 없이 썼다. 이번엔 타자 검증이 잡았지만, **Context
  단계에서 `inspect.signature`로 선확인**했어야.
- **카탈로그-동작 불일치를 오래 방치**: 4종 데모 카탈로그(007부터)가 실제 2 메커니즘과 어긋난 채
  여러 스펙을 지나왔다. "보이는 토글이 실제로 뭘 하는가"를 더 일찍 1:1로 맞췄어야.

## 다음에 다르게 / 추후
- 인기 외부 라이브러리는 **가치 명제(공식 문서·사례)부터** 잡고 소스로 검증 — 소스만 보면 부분 구현을
  전체로 오판한다.
- 외부 함수 호출 인자는 **`inspect.signature`/소스로 키워드까지 선확인**(Context 단계 체크리스트화 고려).
- **agent 축 메모리**: 안전한 쓰기 채널(관리자 저작 UI 등) 설계 후 후속 스펙. 시그니처 슬롯은 예약됨.
- **라이브 통합 테스트(사용자 브랜치)**: 실제 AND 필터 회상·user↔session 격리·교차세션 유저 영속·
  실 DB 마이그레이션 up/down·임베딩 다운 graceful. 010처럼 검증용 임시 스코프+정리를 처음부터 설계.
