# 020 — 회고: 에이전트 전용 메모리(agent_id) 자가기록 + 관리자 큐레이션 (스펙 029)

날짜: 2026-06-26 · 브랜치 `feat/agent-service` · 커밋만(푸시/머지 안 함)

## 무엇을 했나

020이 미룬 `agent_id` 축을 A~E 한 스펙으로 채웠다: 회상 배선(chat.py 스코프 분리),
자가기록 도구(`save_agent_knowledge`, agent_id-only·infer=False), mem0 CRUD 헬퍼,
관리자 CRUD API 4개, Admin "에이전트 지식" 패널.

## 무엇이 잘 됐나

- **스코프 분리가 자동 누출을 닫았다.** `add_scope`(user_id+run_id)와
  `recall_scope={**add_scope, agent_id}`를 분리 → 자동 턴-add는 절대 agent_id를 안 단다.
  정적/라이브 모두 검증. 020의 "쓰기 축 = 소유자" 원칙을 코드 불변식으로 고정.
- **라이브가 단위테스트가 못 본 걸 잡았다.** 격리 단위테스트는 26개 다 PASS였지만,
  라이브에서 (1) LLM이 "기억해줘"에 자가기록 도구를 호출해 PII를 agent_id에 쓰는 경로,
  (2) 비판리뷰가 교차-에이전트 id 변조를 드러냈다. **"통과 = 안전" 아님**을 다시 확인.

## 무엇이 어긋났고 무엇을 배웠나

1. **의도적 쓰기 채널이 닫았던 누출을 되연다.** 패시브 스코프-태깅(020)은 자동추출 누출을
   막았지만, 자가기록 *도구*를 열자 LLM이 사용자의 "기억해줘"에 복종해 PII를 agent_id에
   넣었다. 도구 프롬프트의 "유저 정보 저장 금지"는 **격리 경계가 아니다**(사용자 지시가 이긴다).
   → 사용자 결정: 현행 유지(즉시 활성 + 사후 삭제), 관리자 큐레이션으로 흡수. **단정하지 말고
   라이브로 한 겹 더 파라**는 원칙이 정확히 작동. learning 031로 자산화.
2. **UI가 자기 id만 노출 ≠ 엔드포인트가 소유권 강제.** 공유 pgvector라 mem0 update/delete는
   전역 id로 동작 → path agent_id 없이는 A의 화면에서 B 행을 변조 가능. 027("프런트 필터는
   백엔드 가드가 아니다")의 재현. `_assert_owns`로 백엔드에서 강제, 라이브 404 확인.

## 후속(이번 범위 밖, MED)

- `resolve_agent_mem_cfg`가 `_load_context`의 mem_cfg 해석을 중복 → 공유 리졸버로 추출.
- 관리자 add가 mem0의 생성 id를 버리고 `{ok:true}` 반환 → dedup/0건 케이스 비가시.

## 참고 자산(적용)

- 020(다층 스코프) → agent_id 한 축만 레일에 싣기. 누출 원칙 계승.
- 027(프런트 필터≠백엔드 가드) → 소유권 가드를 엔드포인트에서.
- 030(실 브라우저 검증) → AgentMemoryPanel을 Playflow+시스템 Chrome로 직접 확인.
