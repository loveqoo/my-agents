# 057 — RBAC/소유권 경계 스펙 체크리스트 (회고를 재발방지 훅으로)

## 무엇을 / 왜

066→067→068이 연속으로 *같은 부류*(읽기만 스코핑, 다른 입구 누락)를 사후 봉합했다. learning
069·070이 교훈을 남겼으나 **다음 스펙 설계 때 상기를 강제하는 구조가 없었다**. 회고는 다음 작업
Context에서 상기·적용될 때만 복리가 된다 — 그 상기를 *강제*하는 훅을 만들었다(spec 069).

## 핵심 결정

1. **위치 = 폴더 컨텍스트 `docs/spec/CLAUDE.md`** (INDEX 한 줄도, 루트 CLAUDE.md도 아님). 폴더
   컨텍스트는 스펙 폴더 작업 시 스코프로 *자동 로드*되므로, INDEX 한 줄(Context에서 *골라야* 보임)보다
   강제력이 한 단계 높다. "고르기"가 아니라 "눈앞에 옴". — 훅의 강제력은 *검사지점이 발생지점과
   같은 스코프에 로드되느냐*로 갈린다.
2. **체크리스트 6항목** = 입구 열거 / 입구별 소유권(읽기 SELECT-WHERE·쓰기 무덮어쓰기·생성 스탬프) /
   단일 헬퍼 / 404 비노출 / 검증 사다리 3런 / 자가-잠금 핀. 전부 출처 learning(068/069/070)을 가리켜
   불충분 시 full로 갈 경로 보존.

## 가장 큰 교훈 — 체크리스트조차 적대검증 대상

작성 직후 적대 타자에 "066~068을 *사전에* 잡았겠는가 + 놓친 부류"를 던졌다. 과거 4개는 매핑 통과,
그런데 **미래 부류 P1 갭 2개**가 나왔다: (1) 입구 열거에 `delete` 누락(실코드 `memory_routes.py:170`·
`agents.py:613`에 소유권 걸린 삭제 실재), (2) mem0(전역 id·벡터DB)는 SELECT-WHERE 불가 →
`_assert_user_owns`→`delete_memory` **check-then-act TOCTOU**. 즉 *"모든 입구를 세라"는 체크리스트가
정작 입구 일부를 빠뜨린* 자기모순. "입구 열거"가 닫힌 집합이 아니었다.

이게 메타적으로 의미심장하다: 방어 자산(체크리스트·가드 규칙)을 만들었다고 끝이 아니라, **그 자산
자체를 적대검증에 걸어야** 한다. happy-path(과거 결함 매핑)는 초록인데 여집합(미래 부류)에서 P1이
나오는 패턴은 068 D6(라이브 12/12 초록 뒤 codex가 자가잠금 발견)와 *동형*이다 — 가드를 봐도
가드의 여집합을 봐야 한다(installed-guard-isnt-covering-guard).

## 반영

P1 2개 + A2A 외부입구(P2) + fetch-then-check 잔존(P2) + 트리거 자가판정(P3)을 항목 1·2·트리거에
흡수. 트리거는 "작성자 직감"에서 *객관 신호*(user_id·테넌트 컬럼·소유권 헬퍼 접촉)로 바꿨다 —
자가판정에 맡기면 "경계인 줄 몰라서 빠뜨린" 066의 맹점이 트리거 단계로 옮겨갈 뿐이라서.

## 잔여

- fetch-then-check 타이밍 오라클(retro 056 P3-1)은 여전히 *코드*엔 남음 — 체크리스트는 신규 스펙을
  가드하고 기존 잔존은 "명시 기록"으로만 가시화. 067 영역 정제는 별도 작업.
- 이식 가능 원리는 learning 072(메타 가드의 열거는 닫힌 집합이 아니다).
