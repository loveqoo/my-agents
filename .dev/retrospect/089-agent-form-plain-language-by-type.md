# 089 — 에이전트 폼 재구성: 유저 언어 + 종류가 설정을 가른다 (스펙 108)

## 무엇을 / 왜

106·107로 능력 UI를 냈으나 사용자가 **세 번째** 지적: "impl 같은 내부어 쓰지 마라", "MCP가 두
곳에서 설정돼 이해 못 한다", "'첫 매치 위임' 어렵다 — 단어·UX가 직관적이지 못하다." 표면 수정을
반복하다 뿌리를 놓치고 있었다: **내부 아키텍처(impl·브로커·능력 kind)를 그대로 UI 단어로 노출**했고,
런타임은 종류에 따라 다른 것만 쓰는데 폼은 둘 다 보여줘 **구조적 중복**(MCP 두 번)이 생겼다.

## 뿌리 진단 (코드로)

- 직접 응답 에이전트(`DefaultUiAgent`)는 `ctx.tools`(mcps+RAG)를 **직접** 쓰고 broker를 안 씀.
- 조율형(`orchestrate`)은 `ctx.broker`(=capabilities)**만** 쓰고 직접 tools를 안 씀.
- → 두 설정면은 종류에 따라 **상호 배타**. 한 화면에 둘 다 두면 중복은 필연.

## 한 일

- **유저 언어**: "실행 방식 (impl)"→**에이전트 종류**(직접 응답/조율형), "능력 (브로커 위임)"→
  **무엇에 맡길까요?**, kind 라벨 유저어(사용자 기억·문서·도구·다른 에이전트), cap 라벨서 기술
  접미사(`(memory:user)` 등) 제거. "첫 매치 위임" 등 전략 변형 UI서 숨김(기본 전략).
- **종류가 설정면을 가른다**: 직접 응답=직접 자원 칸(메모리·문서·권한·도구), 조율형="무엇에 맡길까요?"
  한 칸. 직접 자원 칸은 조율형에서 숨김 → MCP·문서가 정확히 한 곳에만(중복 소멸).
- 표현 계층만 교체(저장 모델·왕복 무변경). 미사용 된 `/agent-impls` fetch·impls prop 제거(백엔드
  엔드포인트는 유지).

## 잘된 것

- **AskUserQuestion 미리보기로 종류 명명 확정** → 그 단어로 폼 전체를 다시 짬. 유저가 기준 단어를
  직접 골라 추측 제거("직접 응답 / 조율형").
- **뿌리를 코드로 진단**한 뒤 고침 — "MCP가 두 번"이 표면 버그가 아니라 *두 런타임 경로를 한 화면에
  올린* 구조 문제임을 확인(learning 108). 표면만 또 고쳤으면 네 번째 지적이 왔을 것.
- 브라우저 9/9 — **내부어·기술 id 부재를 innerText로 자동 단언**(사람이 놓치기 쉬운 잔여 노출을 측정).

## 배운 것

- **내부 아키텍처를 UI 단어로 노출하지 마라**(learning 108): 시스템의 분해(impl·broker·kind)와
  유저의 멘탈 모델("이 에이전트가 뭘 하나")은 다르다. 같은 자원이 두 설정면에 나오면 *중복을 예쁘게
  정리*할 게 아니라 **모드로 갈라** 한 곳만 남긴다 — 중복은 대개 "두 배타 경로를 한 화면에 올린" 증상.
- **반복되는 UX 지적은 표면 신호**. 세 번 지적받고서야 뿌리(내부어 노출·경로 중복)를 봤다. 두 번째
  지적에서 이미 "왜 또 어렵다 하지?"를 코드로 파고들었어야 했다(feedback: probe-deeper).

## 검증 안 함(사유)

- 백엔드 106 왕복 무변경(표현 계층만) — 기존 verify 유효.
- 메모리 블록 본문의 잔여 용어(historyDepth·mem0·userId)는 **빌딩 블록 콘텐츠**(유저 편집 영역)라 이
  스펙 범위 밖 — 능력 UI 아님.
