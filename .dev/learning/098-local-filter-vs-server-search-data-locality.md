# 098 — 로컬필터 vs 서버검색은 "데이터 지역성"이 가른다 / 사용자 LIKE 입력엔 메타문자 이스케이프

> 스펙 098, retrospect 079. 참고: spec 097(같은 정직성 계약, *반대* 판정)·spec 072/084·learning 097(honesty).

## 배운 것

### 1. 같은 "검색 UX"라도 로컬필터가 정직한지는 데이터가 전량 로드됐는지에 달렸다

097(컬렉션·메모리)에서는 **로컬 텍스트 필터**가 정당했다 — 목록이 이미 전량 클라이언트에 로드돼
있어 부분일치가 전체를 거른다. 098(세션)에서는 **똑같은 로컬필터가 부정직**하다 — 세션은 **서버
페이징**(offset/limit)이라, 로컬필터는 *현재 20행 페이지 안에서만* 걸러 다른 페이지의 매치를
"결과 없음"으로 오인시킨다. 정직성 계약(learning 097: 화면이 "없다"고 말하면 정말 없어야 함)은
같은데 **판정이 정반대**다.

가르는 축은 UI가 아니라 **데이터 지역성**: *전량 클라 로드 → 로컬필터 OK / 서버 페이징 → 서버검색
필수*. 새 검색 UI를 붙일 때 첫 질문은 "이 목록은 전부 여기 있나, 페이지 조각인가?"다. 조각이면
필터는 반드시 서버 파라미터로 내려 페이징 이전 전체 스코프에서 매칭해야 total·빈결과가 정직해진다.

### 2. 사용자 LIKE 입력엔 메타문자 이스케이프가 기본 (covering-guard)

`ilike(f"%{q}%")`는 사용자가 `%`·`_`를 넣으면 그게 **와일드카드로 살아** 과매칭/열거 오라클이 된다
(`a_b` 검색이 `axb`도 잡음). `escape="\\"` + `\`·`%`·`_` 치환(순서 중요: `\` 먼저)으로 리터럴화한다.
happy-path 테스트는 이걸 절대 못 잡는다 — 검증 시드에 *리터럴 `_`를 가진 행과 임의문자 행을 나란히*
심어 "escape 안 하면 둘 다, 하면 하나"를 수치로 갈라야 드러난다(적대 시드 = 보장의 여집합).

## 훅

- 검색 UI를 붙이기 전 "이 목록은 전량 로드인가 서버 페이징인가"를 먼저 답하라. 페이징이면 로컬필터는
  부정직 — 서버 파라미터로 내려 전체 스코프에서 매칭(정직성 계약, learning 097과 동근·반대판정).
- RBAC 스코프가 걸린 목록에 검색을 얹을 땐 predicate를 스코프 WHERE **뒤에 AND**로 — AND는 집합을
  좁히기만 하니 스코프를 넓힐 수 없다(codex도 이 위치를 첫 번째로 확인). 통합 검증은 *타인 데이터와
  공유되는 검색어*(같은 agent_name)로 "스코프를 못 넘음"을 실증하라.
- 사용자 입력이 LIKE로 들어가면 `%_\\` 이스케이프가 기본값. 검증은 리터럴 메타문자 행 + 임의문자 행을
  나란히 심어 수치로 가른다.
