# 035 — 가드는 진실원을 검사하라, 사본 말고 (+ 초록 verify ≠ 견고)

## 맥락
RAG 인제스트(036). pgvector 컬럼은 `vector(RAG_EMBED_DIMS)`로 고정. 인제스트 시 차원 가드를
`len(v) != c.dims`(Collection에 박제한 **사본**)로 검사했다. 내 verify 스크립트는 ✅ ALL PASS.

## 무엇을 배웠나
**1) 3중 가드를 짜도 가드가 진실원(real column width = `RAG_EMBED_DIMS`)이 아니라 그 사본(`c.dims`)을
검사하면 함정이 다시 열린다.** `c.dims`가 컬럼 차원과 어긋나는(drift) 순간, 사본만 통과시키면 잘못된
차원이 DB insert에서 **500**난다 — 막으려던 "조용한 죽음"이 다른 모습으로 부활. 고정 차원의 **단일
출처**를 정해놨으면 검증도 그 출처를 직접 봐야 한다(여기선 `RAG_EMBED_DIMS`와 `c.dims` **둘 다** 검사).

**2) 같은 부류의 결함이 한꺼번에 나왔다 — happy-path 테스트가 못 보는 종류**:
좁은 `except`(예상 예외만 잡아 나머지를 500+미완 상태로 흘림), 비원자 read-modify-write 카운터(동시성
lost update), 무제한 `file.read()`(OOM). 전부 "정상 입력 1건 직렬 실행" 테스트로는 절대 안 걸린다.

**3) 초록 verify 스크립트 = 견고 아님.** 내 `verify_036`는 6개 시나리오 ALL PASS였지만 위 5개 결함을
하나도 못 잡았다. **적대적 서브에이전트 + 타사 모델(codex)을 독립 병렬**로 돌리니 5개에 **수렴**.
두 검증자의 수렴은 거의 확실한 real bug 신호(단독 지적이면 과대평가 의심).

## 어떻게 적용하나
- **함정 가드를 짤 때**: "내가 검사하는 값이 *실제로 깨지는 그 값*인가, 아니면 그 사본/메타데이터인가?"를
  먼저 물어라. 단일 출처를 정했으면 검증도 출처를 직접 본다.
- **verify가 초록이어도 끝이 아니다**: 비자명한 쓰기 경로는 적대적 타자 검증(서브에이전트 1 + codex 1)을
  **병렬**로 받고 **수렴 항목**부터 고친다. happy-path 통과는 바닥선이지 천장이 아니다.
- 점검 대상 체크리스트(이번에 다 걸림): except 폭(예상 외 예외 → 미완 상태 방치?), 카운터/집계의 원자성,
  외부 입력 크기 상한, 외부 응답의 순서/정합 신뢰(index 등), 진실원 vs 사본.
