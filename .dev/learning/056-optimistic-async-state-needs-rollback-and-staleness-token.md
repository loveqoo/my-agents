# 056 — 낙관적 비동기 상태: "도착"은 "유효"가 아니다 — set엔 롤백, 도착물엔 단조 토큰

## 언제
비동기로 데이터를 불러와 화면 상태를 바꾸는 UI에서, 응답을 기다리는 동안 사용자가 다른
선택을 하거나(에이전트 전환·다른 항목 선택·리셋) 그 요청이 실패할 수 있을 때. 특히 응답
도착 *전에* 어떤 상태를 먼저 고정해두는(낙관적) 패턴.

## 무엇을
happy-path 자가 테스트는 "요청→성공 콜백→반영"만 본다. 그래서 두 부류의 결함을 **구조적으로
못 본다**:

1. **stale 도착물**: 더 최신 선택이 이미 일어났는데 오래된 fetch의 성공 콜백이 늦게 도착해
   최신 화면을 덮어쓴다. 성공했으니 catch도 안 탄다 — 조용히 틀린 화면.
2. **실패 후 미롤백**: 응답 *전에* 낙관적으로 고정한 상태(예: 선택된 id)를 fetch 실패 시
   되돌리지 않아, 실패한 대상에 묶인 채 남는다. fetch가 늘 성공하는 테스트는 이 경로를
   절대 안 밟는다.

핵심 명제: **콜백이 "도착했다"는 그 결과가 "아직 유효하다"를 뜻하지 않는다.** 비동기 UI에서
유효성은 도착 시점에 *다시* 판정해야 한다.

## 처방
- **단조 토큰(seq)으로 stale 판정**: 요청 직전 `seq = ++ref.current`로 토큰을 끊고, 콜백
  진입 시 `if (seq !== ref.current) return`. 더 최신 요청·리셋·전환이 토큰을 올렸으면 도착물
  폐기. 리셋/취소 동작도 같은 ref를 bump해 pending을 무효화한다.
- **항상-최신 ref로 도착시점 컨텍스트 일치**: "이 응답이 *지금* 활성인 대상의 것인가"를
  렌더마다 갱신되는 ref(예: `activeExtRef.current = active?.key`)로 도착 시 비교. 클로저에
  잡힌 옛 값이 아니라 현재 값과 맞을 때만 반영.
- **낙관적 set엔 catch 롤백을 짝으로**: fetch 전에 고정했으면 `prev`를 들고 있다가 실패 시
  되돌린다(`prev === undefined`면 키 삭제). set과 롤백은 한 쌍으로 작성 — 따로 추가하지 않는다.
- **대상 좌표가 바뀔 수 있으면 호출시 값으로 고정**: 콜백이 참조할 대상 id는 호출 시점에
  `const target = activeId`로 캡처해, 도중에 활성이 바뀌어도 엉뚱한 칸에 안 쓴다.

## 왜 자가검증이 못 잡나 — 타자가 필요한 이유
이 셋(stale·미롤백·대상 이동)은 전부 happy-path의 *여집합*이다. 내가 상상한 성공 시나리오를
확인하는 자가 테스트로는 안 나온다. 스펙 055에서 codex 적대 리뷰가 1차에 레이스(HIGH)를,
2차 재리뷰에 **롤백 누락(새 버그)**을 짚었다 — "보장 목록의 여집합을 시켜라"(adversarial)의
전형. [[adversarial-review-before-destructive-ship]]·[[verification-ladder-three-rungs]]의 비동기판.
