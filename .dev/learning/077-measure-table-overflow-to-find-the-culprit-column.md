# 077 — "잘려 안 보임"은 추측으로 폭을 조절하지 말고 scrollWidth로 *범인 컬럼*을 측정해 특정하라

## 상황
사용자가 RAG 컬렉션의 "검색" 버튼을 못 찾았다("안 보여서 몰랐어"). 표면 진단은 두 가지가 떠올랐다:
(1) 검색이 흐릿한 `type="text"` 버튼이라 저발견성, (2) 액션 컬럼(width 290)이 넓어 잘림. 첫
직감은 "액션 컬럼이 범인"이었다. 액션을 아이콘으로 압축(290→170)했는데도 **여전히 점검·삭제가
잘렸다**. 추측으로 폭을 더 줄이는 반복 대신 *측정*했다: `table.scrollWidth` vs 래퍼 `clientWidth`로
오버플로우(69px), 각 `th.getBoundingClientRect().width`, 액션 버튼 boundingBox의 `inView`.

## 배운 것 (일반화)
- **레이아웃이 "잘려 안 보임"이면, 잘린 *요소*가 아니라 전체 폭과 *각 칸의 실측 폭*을 재라.** 잘린
  건 우측 액션이지만, 측정은 **범인이 이름 컬럼(392px = maxWidth 360 + 셀패딩 32)**임을 가리켰다.
  내 첫 추측(액션 290)은 틀렸다 — `max-content` 테이블에서 한 칸이 비대하면 *다른 칸*이 스크롤 밖으로
  밀린다. 잘린 칸을 줄여도 안 풀리고, **비대한 칸**을 줄여야 풀린다. 어느 칸인지는 측정만이 안다.
- **측정 지표를 정확히**: `scrollWidth − clientWidth`(오버플로우 px), 컬럼별 `boundingRect.width`,
  대상 버튼 `box.x+box.width ≤ viewport`(inView 불리언). "대충 더 줄여보자"가 아니라 "이름 360→260이면
  셀 100px↓ → OVERFLOW 69→0" 처럼 *계산하고 적용*. 적용 후 재측정으로 0 확인(2회 추측 대신 1회 측정).
- **표면 원인과 진짜 원인을 함께 고친다**: 사용자가 말한 건 "검색이 안 보임"(발견성)이나, 측정이 드러낸
  더 깊은 원인은 *레이아웃 오버플로우*였다. 검색 primary 승격(발견성)만 했으면 1280에서 여전히 잘렸다.

## 어떻게 적용하나
UI 요소가 "안 보인다/잘린다"는 보고를 받으면: ① 브라우저로 실제 화면 캡처(verify-ui-in-browser),
② 잘린 요소를 줄이기 전에 **컨테이너 vs 콘텐츠 폭과 각 칸 폭을 측정**해 *비대한 칸*을 특정,
③ 줄일 양을 계산→적용→재측정으로 OVERFLOW=0 확인, ④ 표면(발견성)과 구조(오버플로우)를 같이 처리.
일회용이라도 측정 스크립트(`measure-collections-table.mjs`)를 남기면 폭 회귀를 싸게 재검증한다.
