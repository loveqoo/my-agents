# 103 — 읽기전용 능력은 정책은 받고 승인은 안 받는다; 교차입구 불변식은 공유 코어에

능력 브로커에 RAG provider(kind=rag, 읽기전용)를 붙이며 두 가지가 선명해졌다.

## 1. 두 게이트를 분리해 두면 읽기전용 provider가 공짜에 가깝다

브로커에는 게이트가 **둘**이고, 둘은 별개다:

- **정책 게이트**(allowlist∩RBAC·deny-by-default·존재비노출): **모든** provider에 적용. 읽기·쓰기
  무관. "이 유저가 이 능력을 볼/부를 자격이 있나."
- **승인 게이트**(`approval_for` → HIL interrupt): **부수효과가 있을 때만** 발화. "이 비가역 동작을
  실행 전에 사람이 확인해야 하나."

이 둘을 처음부터 분리해 뒀기 때문에(101에서 per-provider `approval_for` 도입), 읽기전용 provider는
`approval_for`가 **항상 None** 한 줄이면 HIL 기계 전부를 우회한다 — 그러나 정책은 **완전히** 적용된다.
만약 "읽기전용이니 게이트 생략"으로 뭉갰다면 정책까지 새어 무인가 검색이 열렸을 것이다.

**교훈:** provider 시임을 설계할 때 "정책"과 "승인"을 **한 판정으로 합치지 마라**. 합치면 읽기전용
kind가 등장하는 순간 둘 중 하나를 잘못 끈다(정책을 끄면 보안 구멍, 승인을 억지로 켜면 읽기에도 HIL).
분리돼 있으면 각 kind가 자기에 맞는 게이트만 취한다: rag=정책O/승인X, mcp=정책O/승인(위험툴만).
memory read=정책O/승인X, memory write=정책O/승인O — 확장이 조합으로 떨어진다.

## 2. 여러 입구가 한 코어를 공유하면, 입력 불변식은 코어에 둔다

RAG 검색 코어(`search_collections`)를 **세 입구**가 부른다: 시험 엔드포인트·인-챗 도구·브로커 위임.
codex가 짚었다 — 엔드포인트만 스키마에서 질의 4000자 상한이 있고, 도구·브로커는 무제한이라 거대
질의 1건으로 임베딩 provider를 장시간 점유할 수 있었다.

고칠 자리는 세 입구가 아니라 **코어 한 곳**이다. `search_collections`에 4000자 상한을 1회 넣으니
엔드포인트는 이미 ≤4000이라 무영향, 도구·브로커가 함께 경계를 얻었다. 입구마다 막으면 (a) 드리프트
(한 입구는 4000, 다른 입구는 다른 값/누락)와 (b) 새 입구가 생길 때마다 재발한다.

**교훈:** N개 입구가 공유 코어를 부르면, **자원 경계(길이·크기·top_k 상한)도 그 코어에 둔다**.
스키마 검증은 그 입구에만 걸리는 국소 방어일 뿐, 공유 코어를 부르는 다른 입구에는 안 미친다.
([[cap-the-raw-source-not-the-buffer]]의 "누적 지점에 걸어라"와 같은 결 — 여기선 "공유 지점에
걸어라". 072가 검색 코어를 추출한 바로 그 drift-0 논리의 연장.)

## 적용

- 새 provider kind 추가 시: 정책은 브로커가, 승인은 `approval_for`가 — 절대 한 판정에 섞지 마라.
  읽기전용이면 `approval_for=None`, 정책은 그대로.
- 공유 코어를 부르는 입구가 2개 이상이면, 입력 상한을 **코어에** 두고 입구 스키마 검증은 국소 보강으로만.
- happy-path 초록(단위+통합)은 무제한 입력·빈 id 같은 여집합을 못 본다 — 적대 타자(codex)가 잡는다.
