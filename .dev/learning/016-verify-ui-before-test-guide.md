# 016 — 테스트 가이드는 실제 UI를 확인하고 쓴다

날짜: 2026-06-24
맥락: 사용자가 "집에 왔다, 뭘 테스트해야 하나" → 내가 테스트 가이드 작성.

## 무엇이 잘못됐나
가이드에 "에이전트 폼에서 **시스템 프롬프트를 직접 입력**"이라고 적었다.
하지만 실제 `AgentForm`은 **페르소나 드롭다운**(라이브러리에서 이름 선택)이었다.
free-text 입력란이 아니다. 코드를 안 보고 기억/추정으로 가이드를 쓴 결과.

사용자가 바로 "페르소나를 등록/수정할 메뉴가 없어"라고 막혔다 — 드롭다운엔
시드 페르소나만 있고, 작성 UI(빌딩 블록 persona 탭)는 빈 껍데기 생성 +
죽은 "편집" 버튼이라 새 페르소나를 못 만들었기 때문.

## 배운 것
- **사용자에게 조작 가이드를 주기 전에 해당 화면의 코드를 실제로 읽어라.**
  특히 "어디에 무엇을 입력"류 안내는 폼 컴포넌트를 직접 확인.
- 결과적으로 진짜 빈틈(페르소나 작성 UI 부재)을 발견 → 014로 수정했으니
  나쁘지만은 않았지만, 가이드 자체는 틀렸던 게 사실.

## 아키텍처 메모 (재확인)
페르소나 흐름: 폼 드롭다운(이름) → 백엔드 `resolve_persona(name)`가 본문 조회
→ `agent.persona`(서빙용 해석 본문)에 저장 → 채팅이 시스템 프롬프트로 사용.
즉 "단순 에이전트"의 말투/역할은 **페르소나 본문**이 결정한다.
