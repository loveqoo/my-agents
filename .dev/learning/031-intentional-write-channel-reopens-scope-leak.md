# 031 — 의도적 쓰기 채널은 스코프-태깅이 닫은 누출을 되연다

## 맥락

mem0 다층 스코프(020)에서 누출 차단의 핵심은 **"쓰기 축 = 사실의 소유자"** — 유저 턴은
user_id+run_id로만 태깅하고 절대 agent_id를 안 단다. 그래야 user A의 사적 사실이
"모든 유저에 회상되는" agent_id 메모리로 새지 않는다. 이건 *자동* 경로(매 턴 infer add)에서
패시브하게 보장된다.

스펙 029에서 에이전트가 스스로 일반 지식을 agent_id에 적는 **자가기록 도구**를 열었다.
도구 설명에 "재사용 가능한 일반 지식만, 특정 유저 PII 금지"를 명시했다.

## 배운 것

**LLM은 사용자의 "이거 기억해줘"를 도구 프롬프트의 금지보다 우선한다.** 라이브 테스트에서
유저가 알레르기 정보를 "기억해줘"라 하자 qwen이 곧장 `save_agent_knowledge`를 호출해
그 PII를 agent_id에 저장했고, 다른 유저 세션에서 회상됐다 — 020이 막으려던 바로 그 누출이
의도적 쓰기 채널로 재유입됐다.

**도구 프롬프트의 규율은 격리 경계가 아니다.** 패시브 스코프-태깅(코드가 축을 안 붙임)은
강한 불변식이지만, 모델 재량에 맡긴 "쓰지 마"는 사용자 지시 한 마디에 뚫린다.

## 어떻게 적용하나

- **자동 경로와 의도 경로를 분리해서 검증·서술하라.** 자동 경로는 코드 불변식(축 미태깅)으로
  0-누출을 보장하고 테스트한다. 의도 경로는 "모델이 규율을 어길 수 있다"를 전제로 위험을
  명시하고, **사람이 받아들이거나(사후 큐레이션) 구조로 막아라**(비회상 제안 풀 / 승인 게이트).
- 진짜 보장이 필요하면 프롬프트가 아니라 **구조**로: 자가기록을 회상되지 않는 풀에 격리하고
  관리자가 승격, 또는 PII 분류기로 차단. 프롬프트는 완화이지 경계가 아니다.
- **단정 전에 라이브로 한 겹 더 파라.** 격리 단위테스트 26개 PASS여도 라이브가 누출을 잡았다.
  내 측정이 "안전"이라 말해도 실제 호출 한 번이 반증할 수 있다. ([[probe-deeper-before-concluding]])
