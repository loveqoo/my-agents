# 068 — 컬렉션을 스코핑해도 항목 엔드포인트는 안 스코핑된다: 404-vs-403 열거 오라클

## 언제
멀티테넌트/RBAC CRUD에서 **목록(list) 엔드포인트는 `WHERE owner = me`로 스코핑**하면서, 같은
리소스의 **항목 엔드포인트(`/{id}` GET·resolve·delete 등)는 인가만 걸 때**. 특히 항목 엔드포인트가
"행 없음→404"와 "있지만 권한 없음→403"을 *다르게* 응답할 때.

## 명제
컬렉션 뷰의 스코핑은 **항목 엔드포인트로 자동 전파되지 않는다.** 목록이 타인/숨김 행을 안 보여줘도,
항목 엔드포인트가 404(부재)와 403(존재하나 금지)을 가르면 공격자는 id를 추측해 *목록이 숨긴 바로 그
행들의 존재*를 캐낼 수 있다 — **열거 오라클**. 즉 "리스트는 스코핑됐으니 안전"은 거짓: 스코핑은
*그 엔드포인트 하나*의 가시성일 뿐, 같은 데이터의 다른 입구(item)는 따로 막아야 한다.

핵심: **403은 "이 리소스가 존재함"을 누설한다.** 권한 없는 사용자에게 "여긴 있지만 너는 안 돼"는
이미 존재 정보다. 목록에서 숨긴 행이라면 item 엔드포인트에서도 *부재와 똑같이*(404) 응답해야 은폐가
경로 전체에서 일관된다.

## 표본 (스펙 066)
`Approval` resolve 인가 3-way. `list_approvals`는 `_own_scope`로 비-admin에게 타인·NULL-owner 행을
숨겼다. 그런데 `resolve_approval` 초안은 비가시 행도 `_may_resolve`=False → **403**. codex 적대
리뷰[Low#1]: member가 `POST /approvals/{candidate}/resolve`를 추측 호출하면 404=행 없음, 403=행
있으나 내 것 아님 → 목록이 숨긴 타인 승인 행의 *존재*가 샌다(approval_id가 고엔트로피라 실용 위험은
Low지만 경계는 진짜 새는 것).

처방: `_may_resolve` *앞에* 가시성 게이트 — 비-admin(`_own_scope`≠None)이 못 보는 행(자기 것 아님)은
부재와 동일하게 **404**. 단 *자기* 행이지만 권한 미달(민감 perm)은 **403** 유지(이미 자기 목록에 보여
존재는 알려진 상태 — 은폐할 게 없고, 403이 정확한 "보이지만 권한 부족" 신호).

## 처방 (이식)
1. **list에 스코프를 거는 순간, 같은 스코프를 item 엔드포인트에도 동시에 설계하라.** 둘은 같은
   데이터의 두 입구다. 한쪽만 스코핑하면 다른 쪽이 오라클.
2. **응답 코드로 가시성을 가르라**: "사용자가 *볼 수 없는* 행(목록에서 숨긴 행)"은 item 엔드포인트에서도
   404(부재와 동일). "*볼 수 있으나 행위 권한만 없는* 행"은 403. 판별 기준 = "이 행이 그 사용자의 목록에
   떴는가?"
3. **404-vs-403 분기를 의식적으로 설계하라.** 기본 ORM 패턴(`get_or_404` 후 인가 403)은 자연스럽게
   오라클을 만든다 — 인가 거부를 무조건 403으로 던지면 부재와 갈린다. 비가시 행은 404로 접어라.
4. **위협 모델에 "열거/존재 누설" 축을 처음부터 넣어라.** happy-path·정상 흐름에선 안 보이고(목록이
   이미 스코핑) 적대자가 id 추측 시나리오를 던져야 드러난다 → 자가검증 초록, 적대 rung만 잡음.

## 판별 질문
- "이 리소스의 **목록을 스코핑**했나? 그럼 같은 리소스의 **항목 엔드포인트도** 같은 스코프로 막았나?"
- "항목 엔드포인트가 **404(부재)와 403(금지)을 다르게** 응답하나? 그 차이가 *목록이 숨긴* 행의 존재를
  누설하나?"
- "이 403이 '존재함'을 누설하는데, 그 사용자는 이 행이 존재함을 *원래 알아도 되는* 사용자인가(자기
  목록에 떴나)? 아니면 부재(404)로 접어야 하나?"

## 가족
- **installed-guard-isnt-covering-guard**(가드 설치≠적용 범위 전체) — 여기선 *스코핑*이 한 엔드포인트만
  덮고 형제 엔드포인트는 안 덮음. 같은 "한 지점 방어가 전 경로를 덮는다는 착각".
- **065/066 경계 한 홉 결함**(error-surfacing·normalize-before-guard) — 계약/불변식이 *경로 전체*가
  아니라 한 홉에만 적용되면 다른 홉에서 깨진다는 공통 뿌리. list↔item은 같은 데이터의 두 홉.
- **verification-ladder / adversarial-before-destructive** — 열거 오라클은 단위·라이브 happy-path가 다
  초록인데 적대 codex만 잡았다(존재 누설은 정상 사용자 흐름엔 없음).
