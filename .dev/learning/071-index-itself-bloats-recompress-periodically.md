# 071 — 인덱스도 비대해진다: 한 줄 후크 규약을 어기면 인덱스가 다시 통독 대상으로 회귀

## 한 줄
통독을 줄이려 만든 한 줄 인덱스도, 항목마다 서사를 욱여넣으면 *그 자체가 통독 대상*이 된다 —
인덱스는 *고르게 하는* 한 줄(후크+키워드)로 유지하고, 비대해지면 **주기적으로 재압축**한다(인덱스의 인덱스화).

## 맥락
- 인덱스 층(스펙 043 / [index-layer-for-context-recall])은 "자산이 쌓여 통독이 무거우면 한 줄
  인덱스를 두고 인덱스 먼저, 후크 불충분 시만 full"로 Context 읽기 비용을 95% 낮춘다.
- 그런데 인덱스 줄 자체가 자라면 그 절감이 되감긴다. my-agents 사례: 인덱스 도입 후 추가된
  항목들(특히 적대 검증 회고들)이 한 줄에 명제·표본·처방·판별·가족관계를 다 담아 **평균 585자/줄,
  최장 2686자**까지 비대화 → 인덱스 3종 합 103KB. "한 줄"이 사실상 *요약본*이 되어 다시 통독 대상.

## 무엇을 했나 (2026-06-30)
- **측정 먼저**(추측 금지): always-on(CLAUDE.md+MEMORY.md)=14KB로 작음, on-demand INDEX 3종=103KB·
  평균 585자/줄이 진짜 범인. "큰 것"의 위치를 데이터로 특정.
- **서브에이전트 3개 병렬 위임**으로 각 INDEX를 한 줄 후크로 재압축(메인 컨텍스트 증가 차단, 결과는
  크기만 회수). 103KB→53KB(48%↓), 평균 585→290자/줄.
- **보존을 측정으로 검증**(자가검증 지양): git 원본 대비 항목 번호 70/56/68 전부 보존, 키워드 배열
  멀티셋 diff=서사 속 본문 대괄호 2개(`[0,300]`·`[me]`)만 빠짐(실제 키워드 손실 0).

## 다음에 이렇게
1. **인덱스 줄에 서사를 넣지 마라.** 후크 = "언제 이 항목을 열까" 한 문장 + 키워드. 표본·처방·
   판별·가족관계는 full 파일의 몫. 인덱스는 *고르는* 도구이지 *읽는* 도구가 아니다.
2. **새 자산을 추가할 때 줄을 짧게** 쓴다(같은 턴 인덱스 한 줄 규칙 + 길이 절제). 비대화는 한 번에
   오지 않고 항목마다 조금씩 쌓인다 — 추가 시점에 막는 게 가장 싸다.
3. **주기적 재압축**을 Scaffolding 점검 항목으로 둔다. "컨텍스트가 무겁다" 신호 = 인덱스 줄 평균
   길이부터 측정. 대량 변환은 서브에이전트 위임, 보존은 git 대조 측정.
4. **두 문제를 구분**한다: "이 대화가 큼"(작업 누적 → 새 세션만 즉시 비움) vs "미래 Context가 비쌈"
   (인덱스 비대 → 다이어트가 복리). 자산 압축은 *이미 로드된* 현재 대화를 줄이지 못한다.
