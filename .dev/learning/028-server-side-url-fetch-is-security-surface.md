# 028 — 서버측 외부 URL fetch는 "보조도구"여도 보안 표면이다

날짜: 2026-06-26
출처: 스펙 [026](../../docs/spec/026-external-agent-a2a-card-registration.md) 타자 검증(codex GATE)
연결: [[017-external-agent-a2a-card-registration]], [[027-frontend-filter-is-not-a-backend-guard]],
[[probe-deeper-before-concluding]]

## 교훈

사용자가 준 URL을 **서버가 fetch**하는 기능은, 아무리 "auxiliary/가벼운" 기능이어도 보안 표면을
가진다. 최소 가드는 기능 가치와 무관하게 **기본값**으로 켠다:

- **응답 크기 상한** — `resp.json()`/`resp.content`는 본문을 통째 버퍼한다. 스트리밍으로 읽어 상한
  초과 시 중단(악성·오작동 서버의 메모리·시간 소진 차단).
- **타임아웃** — 연결·읽기 모두.
- **스킴·타입 검증** — http(s)만, 파싱한 JSON의 필드 **타입까지**(`"skills":"x"`가 truthy로 통과하지
  않게).
- **SSRF(내부망 접근)** — 이건 기본값이 아니라 **신뢰경계 판단**. 사설/loopback/link-local 차단은
  보안을 높이지만 dev mock(localhost)·정당한 사내 호출을 깰 수 있어 트레이드오프다.

## 더 큰 패턴 — 타자 지적은 "공짜 수정"과 "결정사안"으로 가른다

codex/서브에이전트가 N건을 지적하면 일괄 "다 고친다" 또는 "보조기능이니 무시"로 처리하지 말 것.

- **공짜 수정**(트레이드오프 없음, 명백히 옳음): 즉시 적용. 예) 크기 상한, 타입 검증.
- **진짜 트레이드오프**(고치면 다른 게 깨짐): 사용자에게 올려 합의. 예) SSRF 차단 ↔ dev mock 깨짐.
  결정은 스펙에 "알려진 한계 → 2차 보강"으로 **기록**해 다음 작업의 Context에서 상기되게 한다.

자가 단정([[probe-deeper-before-concluding]])도 위험하지만, 타자 지적의 **무비판 수용**도 위험하다 —
멀쩡한 dev 흐름을 깨거나 보조도구를 과보강(gold-plating)할 수 있다.

## 적용 방법

- 서버가 외부 URL을 fetch하는 신규 코드를 짤 때: 크기상한·타임아웃·스킴/타입 검증을 **첫 구현부터**
  넣는다(나중 보강 X). SSRF는 "신뢰경계가 어디냐"를 명시적으로 판단·기록한다.
- 타자(codex/서브에이전트) GATE FAIL을 받으면: 각 건을 "공짜/트레이드오프"로 라벨링한 뒤 처리.
  트레이드오프는 사람 결정 + 스펙 기록.
