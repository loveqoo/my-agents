# 069 — 읽기 경로를 스코핑해도, 같은 데이터의 *쓰기/resume 입구*가 소유권을 안 걸면 스코핑이 무력화된다

## 언제 적용
멀티테넌트/RBAC에서 어떤 자원의 *읽기 엔드포인트*(list/detail/messages)에 유저별 스코핑을
새로 거는데, **같은 행을 생성·변경·resume하는 다른 입구**가 따로 있을 때 — 특히 그 쓰기 입구가
`row.owner = current_user`처럼 소유자를 *덮어쓰거나*, 소유자 무검사로 행을 바인딩할 때.
list-vs-item 오라클(068)의 한 단계 위 — 거기선 *두 읽기 입구*의 불일치였고, 여기선 *읽기 게이트 vs
쓰기 입구*의 불일치다.

## 명제
"이 자원을 스코핑했다"는 **그 엔드포인트군의 가시성**일 뿐, 데이터의 *모든 입구*가 아니다.
읽기 게이트(`WHERE user_id==me`, 비가시행 404)가 완벽해도, 같은 행을 만지는 **쓰기/resume 경로**가
(a) 소유자를 무검사로 바인딩하면 *존재 오라클*이 되고, (b) `if user_id: row.user_id = user_id`처럼
소유자를 무조건 덮어쓰면 **소유권 탈취**가 되어, 탈취 후엔 읽기 게이트가 "본인 소유"로 통과한다.
즉 *쓰기 입구가 읽기 게이트의 전제(소유권 진실성)를 깬다*. 읽기 스코핑은 그대로인데, 게이트가
판정하는 `row.user_id`를 공격자가 쓰기 입구로 바꿀 수 있으면 게이트는 우회된다.

## 표본 (스펙 067)
`sessions.py`에 067이 `/sessions` 읽기 스코핑(list 본인만·item 비가시행 404·배지 본인집계)을 정확히
걸고 4 rung 통과. 그러나 codex 적대 rung이 chat resume(`POST /agents/{id}/chat`)를 짚음:
- `_load_context`(`chat.py:210-220`)가 세션을 `session_id AND agent_pk`로만 바인딩 — 소유자 무검사.
- SSE 첫 프레임이 `ctx["session_id"]`를 에코 → 추측 적중 id가 *새 id*(`sess-`+`token_hex(3)`=24비트,
  `chat.py:229`)와 구별됨 = **열거 오라클**.
- `_persist`(`chat.py:332-333`) `if user_id: sess.user_id = user_id` → member가 타인 세션 resume 시
  **소유자를 attacker로 덮어씀**. 이후 067이 막 게이트한 `GET /sessions/{id}/messages`가 "본인 소유"로
  통과해 피해자 전사 누출(2차). *모델 컨텍스트는 클라 본문(`chat.py:458,525`)으로 구성 → resume
  응답 자체로는 히스토리 안 샘; 누출은 탈취 후 읽기 게이트 통과로 발생.*
happy-path(자기 세션 resume) 전부 초록 → 적대 rung만 잡음.

## 처방
1. **읽기 스코핑을 설계하는 *순간* 같은 데이터의 쓰기/resume 입구도 같은 표에 올린다.** "이 행을
   list/detail로 *읽는* 경로"뿐 아니라 "*생성·변경·resume*하는 경로"를 열거하고, 각 입구가 소유권을
   어떻게 다루는지 적는다. 위협모델 축: "같은 데이터의 다른 입구(write/resume)".
2. **쓰기 입구는 소유권 *이전*을 기본 금지(불변식).** 기존 non-null 소유자를 *다른* 유저로 덮어쓰지
   않는다(`row.user_id`가 이미 있고 다르면 거부/무시). 소유권 부여는 *생성 시 1회*로 한정.
3. **resume/조회 바인딩에 소유자 스코프 주입.** 비-admin은 `WHERE owner == me`로만 resume, 타인/NULL은
   *새로 만든 것과 구별 안 되게* 부재 취급(존재 오라클 차단 — 068의 404 은폐를 쓰기 입구에도 적용).
4. **추측 가능한 식별자엔 충분한 엔트로피.** 오라클이 존재하면 24비트는 brute-force 사정거리.
   소유권/가시성 게이트는 엔트로피에 기대지 말되, 엔트로피도 같이 올린다(방어다중화).

## 판별 질문
- "이 자원의 읽기를 스코핑하면서, 같은 행을 *만드는/바꾸는/resume하는* 다른 엔드포인트를 열거했나?"
- "그 쓰기 입구가 `row.owner = me`로 소유자를 덮어쓰나? 그러면 읽기 게이트의 전제가 깨진다."
- "쓰기 입구가 소유자 무검사로 행을 바인딩하나? 그러면 존재 오라클이 된다."

## 가족
- **068**(list-vs-item 열거오라클) — 거긴 *두 읽기 입구*의 코드 불일치, 여긴 *읽기 게이트 vs 쓰기
  입구*의 불일치(같은 데이터, 한 입구가 게이트의 전제를 깸).
- **installed-guard-isnt-covering-guard** — 가드 설치≠적용범위 전체 덮음; 여긴 읽기 게이트가 쓰기
  입구를 안 덮어 그 틈으로 게이트 판정값(owner)이 변조됨.
- **move-breaks-references-both-directions** — 한 방향(읽기)만 보면 green이지만 다른 방향(쓰기)이 깸.
- **verification-ladder-three-rungs / adversarial-review-before-destructive-ship** — 단위·라이브·브라우저
  happy 초록인데 적대 codex만 "보장 목록의 *여집합*(같은 데이터의 다른 입구)"을 던져 잡음.
