# 055 — 새 아웃바운드 경로는 네트워크 가드를 자동 상속하지 않는다 (의존성은 위험 기본값을 들고 온다)

레포가 기존 경로에서 SSRF/리다이렉트 방어를 해왔더라도, **새 아웃바운드 HTTP 경로**(새 클라이언트
라이브러리·새 통합)는 그 방어를 *물려받지 않는다*. 더 나쁜 건, 새 의존성이 **위험한 기본값**
(`follow_redirects=True` 같은)을 켠 채로 들어와 가드 통과 *이후*에 경계를 다시 연다는 점이다.

## 신호
- 코드베이스에 이미 SSRF 가드가 있다("우린 `guard_url` 쓴다") → **있다 ≠ 이 경로에 적용된다**.
- 새 라이브러리로 외부에 **나가는** 호출을 추가했다(MCP 클라이언트, webhook, 카드 fetch…).
- 그 가드가 **최초 URL의 IP만** 검사한다(리다이렉트 홉·DNS 리바인딩은 사정권 밖).
- 새 HTTP 클라이언트의 기본이 `follow_redirects=True` / TLS 검증 느슨 / 프록시 신뢰 등.

## 깨지는 방식 (이번 사례)
1. 공인 URL이 `guard_url`(최초 IP만 검사)을 **통과**한다.
2. 서버가 `302`로 사설/메타데이터 대역(`169.254.169.254`, `10.x`, 루프백)으로 **재유도**한다.
3. 새 클라이언트가 기본값대로 리다이렉트를 **추종** → 가드는 이미 끝났다.
4. 그 요청에 **복호화된 Bearer 토큰**까지 실려 나가 내부 표면 + 토큰이 동시에 샌다.
레포의 다른 경로(a2a_client/agent_card)는 `follow_redirects=False`로 막아왔지만, *새* 경로는 안 막혔다.

## 처방
1. **경로마다 가드를 재확인**: "가드가 설치됨"이 아니라 "이 *새 경로*에서 켜졌나"를 묻는다.
   나가는 모든 지점(런타임 실행 + 탐색/테스트 엔드포인트)을 빠짐없이 배선 — 한 곳이라도 빠지면 샌다.
2. **새 의존성의 기본값을 명시적으로 잠근다**: 라이브러리가 주입 훅을 주면(httpx client factory 등)
   그걸로 `follow_redirects=False`·TLS 검증·타임아웃을 *fail-closed*로 고정. 기본값에 기대지 말 것.
3. **회귀가드는 설정값 자체를 단언**: "팩토리가 존재한다"가 아니라 **클라이언트가 실제로
   `follow_redirects is False`** 인지 단언(`installed-guard-isnt-covering-guard`). 그리고 모든
   outbound 지점이 그 팩토리를 *배선*했는지 grep.
4. **위협모델에 리다이렉트·리바인딩 홉을 넣는다**: 최초 IP만 보는 가드는 happy-path(직링크)만
   초록. 적대 타자에게 "공인→3xx→사설 + 토큰 재전송"을 시켜라.

## 메타패턴 위치
`installed-guard-isnt-covering-guard`(가드 검사지점≠부수효과 발생지점)와 `move-breaks-references`
(이동/추가가 양방향 참조를 깸)의 **네트워크 판** — 여기서 더한 축은 **(a) 새 경로는 기존 방어를
상속 안 함, (b) 새 의존성은 위험 기본값을 들고 옴**. 044(가드 경계)·050(연산 대칭)의 친척이되,
좌표가 "코드 위치"가 아니라 "아웃바운드 *경로*"다. 거짓 green(051)과도 인접 — 직링크 통합테스트만
돌리면 리다이렉트 구멍은 초록으로 남는다.
