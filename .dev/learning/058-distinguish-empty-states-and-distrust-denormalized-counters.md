# 058 — 의미가 다른 두 빈 상태는 구분하라, 비정규화 카운터는 행 수의 진실이 아니다

## 언제
선택/로드로 채워지는 화면이 `items.length === 0`만으로 "비어 있음"을 판정할 때.
특히 **데이터를 가리키는 메타 카운터**(`turns`, `count`, 배지 숫자)가 **실제 영속 행 수**와
어긋날 수 있을 때(시드/레거시/마이그레이션 이전 데이터).

## 무엇을
스펙 055 Playground: 사용자가 "세션을 선택해도 **아무것도 안 나온다**, 히스토리가 나와야
하지 않나" 보고. 코드는 정상이었다 — `loadSession`이 `getSessionMessages(sid)`로
`[]`를 받아 `setConvos([])` 했고, DebugChat은 `messages.length===0`이면 새 대화와
**똑같은 프롬프트 카드**를 띄웠다. 즉 *골랐는데 빈* 화면 = *아무것도 안 고른* 화면.

근본은 두 겹이다:
- **비정규화 카운터 ≠ 행 수**: `Session.turns`는 저장된 카운터 컬럼이지 `Message` 행 수의
  집계가 아니다. 시드 세션은 `turns=14`인데 `Message` 0행 → `getSessionMessages`가 `[]`.
  "14턴인데 불러올 게 없다"는 모순은 카운터를 행의 진실로 착각해서 생긴다.
- **빈 상태 모호성**: 의미가 다른 두 상태(새 대화 시작 vs 선택했으나 메시지 없음)가
  픽셀이 같으면, 사용자는 "내 선택이 먹히지 않았다"로 읽는다. 화면은 정확히 동작했는데도.

## 처방
- **빈 상태를 의미별로 분기**한다. 같은 `empty`라도 *선택 맥락의 유무*로 갈라라:
  `pickedButEmpty = empty && !!currentSessionId && !streaming` → "이 세션엔 불러올 메시지가
  없습니다(영속 전/레거시). 입력하면 이 세션에 이어 쌓입니다"로, 새-대화 프롬프트 카드와
  **다른** UI를 띄운다. 빈 화면 자체가 "왜 비었는지 + 다음 동작"을 설명해야 한다.
- **비정규화 카운터를 진실로 쓰지 마라.** 표시·분기에 `turns` 같은 저장 카운터를 믿지 말고,
  실제로 불러온 행(`messages.length`)을 기준으로 상태를 정한다. 카운터와 행이 갈라질 수 있는
  데이터(시드/레거시)를 검증에 반드시 포함한다(learning 057의 데이터 다양성과 같은 뿌리).
- **"동작은 맞는데 사용자가 깨졌다고 느낀다"는 버그다.** loadSession은 옳았다 — 결함은
  *구분되지 않는 빈 상태*라는 UX에 있었다. "코드가 맞다"로 닫지 말고 사용자가 보는 화면이
  의도를 전달하는지까지가 완료다([[probe-deeper-before-concluding]]).
