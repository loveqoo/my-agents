# 108 — 내부 아키텍처를 UI 단어로 노출하지 마라 / 같은 자원이 두 곳에 나오면 모드로 갈라라

## 맥락

능력 브로커 UI(106·107)를 사용자가 세 번 지적: "impl 단어 쓰지 마라", "MCP가 두 곳에서 설정돼
이해 못 한다", "'첫 매치 위임' 어렵다". 표면 수정을 두 번 반복하고서야 뿌리를 봤다.

## 배운 것

### 1. 시스템의 분해 ≠ 유저의 멘탈 모델 — 내부어를 UI로 꺼내지 마라

impl(레지스트리 키)·broker·capability·kind는 **구현이 세상을 나눈 축**이다. 유저는 그렇게 생각하지
않는다 — "이 에이전트가 **뭘 하나**"(직접 답하나, 남에게 넘기나)로 생각한다. UI 단어는 유저의 축을
따라야 한다:
- "실행 방식 (impl)" → **에이전트 종류**(직접 응답 / 조율형)
- "능력 (브로커 위임)" → **무엇에 맡길까요?**
- cap id(`memory:user`·`mcp:...`) → 화면엔 사람이 읽는 이름만, id는 값으로만.

내부어가 UI에 새면 유저는 문서를 읽어야 쓸 수 있다. **innerText에서 금지어(impl·브로커·위임)와 기술
id를 자동 단언**하면 사람이 놓치는 잔여 노출을 측정으로 잡는다(자가검증 회피).

### 2. 같은 자원이 두 설정면에 나오면 = 두 배타 경로를 한 화면에 올린 증상

MCP가 두 곳(직접 도구 칸 + 위임 능력 칸)에 나온 건 표면 버그가 아니었다. 코드로 진단하니:
- 직접 응답 에이전트(`DefaultUiAgent`)는 `ctx.tools`(직접 mcps/RAG)**만** 쓴다.
- 조율형(`orchestrate`)은 `ctx.broker`(capabilities)**만** 쓴다.

런타임은 **종류에 따라 하나만** 쓰는데 폼은 둘 다 보여줬다. → 중복은 필연이었다. 해법은 중복을
*예쁘게 정리*가 아니라 **모드가 설정면을 가르게** 하는 것: 종류=직접 응답이면 직접 자원 칸만,
종류=조율형이면 위임 칸만. 그러면 자원이 정확히 한 곳에만 나온다.

→ **처방**: 폼에서 같은 대상이 두 번 나오면 "두 런타임 경로를 한 화면에 올린 건 아닌가"를 코드로
확인하라. 맞으면 모드(종류)로 갈라 배타 렌더. 배타 경로를 한 화면에 두면 사용자는 "왜 두 번?"을
영영 이해 못 한다.

### 3. 반복되는 UX 지적은 표면 신호 — 두 번째에 뿌리를 파라

세 번 지적받고서야 뿌리(내부어·경로 중복)를 코드로 팠다. 첫 수정(접이식)·둘째 수정(카운트)은 표면만
만졌다. **같은 종류 불만이 두 번 오면** 그건 "더 다듬어라"가 아니라 "틀을 잘못 잡았다"는 신호 —
표면을 또 손대기 전에 아키텍처와 유저 멘탈 모델의 간극을 의심하라(feedback: probe-deeper).

## 부수

- UX 명명은 **AskUserQuestion 미리보기(ASCII 목업)**로 유저가 기준 단어를 직접 고르게 → 추측 제거,
  그 단어로 폼 전체를 재구성.
- 표현 계층만 교체(저장 모델·왕복 무변경) → 회귀 위험이 뷰에 갇힘(learning 107 재확인).
