# 049 — 상태를 지우는 청소는 *재생성 소스*가 살아있으면 무의미하다 — 행이 아니라 발생원을 고쳐라

## 맥락
스펙 050 Phase 2에서 테스트 유저 verify032를 DB에서 삭제했는데 곧 **다시 생겼다**. `ps eww <pid>`로
실행 중 api 서버를 보니 env에 `ADMIN_EMAIL=verify032`가 있어 매 `--reload`마다 seed_admin()이 재생성하고
있었다. 그 env는 어떤 커밋 파일에도 없는 셸 `export`였다 — grep으로는 영원히 안 나온다.

## 메타패턴
파괴적 정리에서 **삭제(행/상태)는 증상이고, 그 상태를 만드는 *발생원*(generator)이 원인**이다. 발생원이
계속 돌면 상태를 지워도 즉시 복원된다. 청소의 단위는 "지금 있는 행"이 아니라 "그 행을 *만드는 주체*"여야 한다.

발생원의 흔한 형태: 부팅 시드(seed_admin), 크론/배치 잡, watcher/리로더, 다른 프로세스의 env·설정,
재시도 큐. 이들은 **DB 밖**에 있어 DB만 보면 안 보인다.

041(WHERE 바이트)·044(WHICH URL)·046(WHICH-span)·047(WHICH-set 완전성)·048(WHEN 수명) 메타패턴 군에
**WHENCE(발생원) 축**을 더한다 — 앞 축들은 "한 작업의 *범위*"를 다뤘고, 이건 "그 상태가 *어디서 다시
오는가*"를 다룬다.

## 적용점
- 파괴적 정리 후 **재발 여부를 반드시 확인**한다. 다시 생기면 행이 아니라 발생원을 찾는다.
- 발생원이 DB에 안 보이면 **프로세스 env·시드·크론·리로더**를 의심한다(`ps eww`로 실행 중 env 확인).
- 고치는 층을 발생원에 맞춘다: env 재시드면 **config층**(.env.example/.env), 시드 코드면 시드 게이트,
  잡이면 잡 설정. 행 삭제는 발생원을 끈 *뒤에야* 영구적이다.
- 살아있는 프로세스의 env는 파일을 고쳐도 안 바뀐다 — **재기동**이 필요하면 액션 아이템으로 명시한다.
- [[self-fixture로 통합 rung을 잃지 않기(045)]]는 이 문제의 *예방판*이다: 테스트가 정크를 안 남기게
  자가정리시키면 애초에 발생원이 안 생긴다.

관련: 045(self-fixture), 037(파괴 노브 바닥), 048(수명 축).
