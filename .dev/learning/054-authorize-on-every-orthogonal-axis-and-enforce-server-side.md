# 054 — 인가는 모든 직교 축에서, 강제는 서버에서

공유 저장소의 행을 (주체) 가 (대상) 의 (행) 에 접근하려 할 때, 가드를 **한 축만** 걸면
다른 축에서 path 위조로 샌다. 격리는 **직교하는 축마다 가드를 두고 합성**해야 성립한다.

## 신호
- 라우트가 `/{user_id}/{mem_id}`처럼 **두 식별자**를 path로 받는다.
- 가드가 그중 하나(주체↔user_id)만 검사한다 → 나머지(user_id↔row)는 무방비.
- 또는 클라에 "넌 어드민이야" 같은 **capability**를 내려주고 그걸로 끝낸 듯 느낀다.

## 깨지는 방식 (이번 사례)
- **주체 가드만**: member가 `DELETE /memory/user/{자기id}/{남의_mem_id}` — path user_id를
  자기로 위조 → 주체 가드 통과 → 남의 행 삭제. (행-소유권 가드가 404로 막아야 함.)
- **행 가드만**(가드 추가 전): 애초에 임의 user_id 열람이 열림.
→ 두 축(주체×대상, 대상×행)을 **둘 다** 걸어야 어느 쪽 위조도 안 통한다.

## 처방
1. **축을 센다**: 요청이 엮는 식별자 쌍마다 가드 하나. path-param 신원 검사는 그 자체로
   spoofable — 반드시 다음 축(소유권)과 합성.
2. **모든 동사에**: GET뿐 아니라 PATCH·DELETE까지 같은 가드(050의 연산 대칭). 데이터
   페치 *이전*에 게이트.
3. **capability는 권고, 강제는 서버**: 프론트에 내린 플래그(드롭다운 노출 등)는 UX 편의일
   뿐, per-route에서 **독립적으로** 재판정(defense in depth). 클라를 무시한 직접 호출도 막힘.
4. **fail-closed 확인**: 인가 의존성이 미초기화/예외일 때 allow가 아니라 deny/500인지 단언.
5. **검증**: 주체 N종 × (self/other) × (동사) 매트릭스를 단위로, 그리고 *실제* 약한-권한
   주체로 통합 1회(쿠키 라운드트립) — happy-path 어드민만으론 경계를 못 잡는다.

## 메타패턴 위치
050(연산=동사 축)·018(가드 범위)의 친척 — 여기서 더한 건 **대상이 두 식별자로 좌표화될 때
축이 둘**이고, 한 축 가드는 다른 축 위조에 무력하다는 점. 053(이 처방 적용)·052(같은
라우터의 신원 보강)·031(RBAC seam, is_superuser 우회/머신토큰 등가).
