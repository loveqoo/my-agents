# 050 — 파괴 헬퍼의 한 연산(delete)에 건 가드는 *권한 등가*인 형제 연산(create)을 안 덮는다

## 맥락
던짐용 super를 만드는 provisioner(`tests/_provision_super.py`)의 `delete`는 keep-list·prefix로 단단히
막아 "실계정·부트스트랩 admin을 절대 못 지운다"고 자신했다. 적대 리뷰가 `create`의 EXISTS 분기를 짚었다:
`create("alice@example.com", ...)` → 이미 있는 실계정 alice를 **is_superuser=True로 승격**시킨다. 가드가
전혀 없었다.

## 메타패턴
파괴 헬퍼가 한 연산만 막고 형제 연산을 열어두면, **그 형제가 같은 등급의 피해를 낼 수 있을 때** 위협이
샌다. 여기서 핵심은 **블래스트 반경의 등가성**이다:
- `delete(실계정)` = 계정 소멸.
- `create(실계정)` = is_superuser=True 승격. 그런데 `is_superuser`는 authz.require가 casbin을 통째로
  우회하는 권한 → **계정을 탈취해 전권을 주는 것**. 소멸과 *다른 방향이지만 같은 등급*의 피해다.

"delete를 막았으니 안전하다"는 delete의 위협 모델만 본 것. 보호 대상(실계정)에 대한 **모든 연산**을 위협
모델에 넣어야 한다. 대칭 위협은 대칭으로 막는다 — 같은 가드(`_disposable`)를 양쪽에 건다.

044("안전장치≠안전, 적용 범위 경계")·048("한 수명 경계≠수명 전체")의 친척이다. 044는 *범위*(한 URL),
048은 *수명*(한 시점), 이건 ***연산***(한 동사) 축이다 — 같은 객체에 대한 sibling 동사를 빠뜨린다.

## 적용점
- 파괴/특권 헬퍼를 만들 때, 보호 대상에 대한 **모든 동사**(create/update/delete/promote)를 나열하고
  *각각*이 보호 불변식을 깰 수 있는지 본다. delete만 가드하고 끝내지 않는다.
- 특히 **권한 상승**(super 승격, 역할 부여)은 삭제만큼 위험하다 — "쓰기"라고 가볍게 보지 않는다.
- 가드는 한 곳에 모으고(`_disposable` 같은 술어) 모든 연산 입구에서 동일하게 호출한다(중복 분기 금지).
- 멱등성("이미 있으면 보장만")이 실계정에 작동하면 *멱등이 곧 에스컬레이션*이다 — 멱등 분기도 가드 안에 둔다.
