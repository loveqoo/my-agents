## 증상

안전 핵심 게이트(승인 전 위험 도구 부수효과 0)를 만들며 검증을 두 겹으로 깔았다: **인프라-경량 단위
게이트 테스트**(실 게이트 코드 + 스텁 모델 + MemorySaver, 16/16)와 **적대 서브에이전트 리뷰**(불변식 여집합
탐색). 둘 다 통과/무사. 그런데 그 위에 **실 DB·실 AsyncPostgresSaver를 태우는 HTTP 통합 probe**를 한 겹 더
깔자, 단위·적대가 구조적으로 못 보던 결함 둘이 나왔다:

1. **배포 상태 drift:** 통합 probe에서 interrupt가 **전혀 안 났다**. 게이트 코드는 완벽했다 — dev DB가
   seed.py에 위험 도구가 추가되기 *전*에 시드돼 있어 도구가 미배선이었다(seed는 빈 테이블일 때만 적용).
   단위 게이트는 도구를 *직접 주입*하니 이 갭이 영원히 안 보인다. 적대 리뷰도 *코드*를 읽지 *실행 중인 DB의
   행*을 읽지 않으니 못 본다. 실 영속 상태를 태우는 rung만이 잡는다.
2. **다중 액터 글루:** event_stream의 stream_mode 튜플 파싱 → interrupt 감지 → Approval row 생성 → 별도
   HTTP 요청(resolve)에서 같은 체크포인트로 재개. 이 *요청 간·프로세스 간* 배선은 한 그래프 인스턴스를
   in-memory로 돌리는 단위 테스트엔 존재하지 않는 면이다.

## 교훈

**검증 사다리의 rung들은 서로의 사각을 덮는다 — 하나로 다른 걸 대체할 수 없다.**

- **단위(시맨틱):** 로직이 맞나. 입력을 직접 주입 → *배선·배포 상태는 가정*.
- **실인프라 통합:** 실제로 배선·영속됐나. 실 DB/체크포인터/HTTP를 태움 → *시드 drift, 요청 간 글루,
  프로세스 경계*를 잡음. 단위가 "직접 주입"으로 우회한 모든 전제가 여기서 검사된다.
- **적대 리뷰:** 불변식의 여집합(상상 못 한 범주, learning 038). *코드*를 읽음 → 인가 홀·동시성·fail-open.

세 rung이 잡는 결함이 **겹치지 않는다**. 부수효과를 막는 게이트엔 셋 다 깔아라.

## 적용 규칙

- 게이트/배선/스키마를 바꾸면 **실 영속 상태를 태우는 통합 rung을 반드시 한 겹** 둔다(목 DB 말고 실 DB).
  "단위 통과 + 적대 무사"는 *배포된 행이 실제로 그 코드 경로를 태우는지*를 증명하지 않는다.
- **시드/설정 변경은 기존 환경에 안 먹는 것을 기본 가정**한다(빈 테이블 가드·`if not exists`). 위험 도구
  배선처럼 안전에 닿는 변경은 idempotent 업서트나 데이터 마이그레이션을 같이 — 통합 테스트가 setup에서
  보정해야 했다면 그게 곧 "프로덕션도 보정 필요" 신호다(스펙 §7 빚으로 명문화).
- 통합 테스트 스텁은 **프로덕션 재구축 경로를 정직히 모사**한다. 재개가 새 모델 인스턴스를 만들면 스텁도
  상태 없이(메시지 내용으로 분기) 동작해야 — idx 카운터 스텁은 재개 시 거짓 통과/무한루프로 샌다.
- 검증 실패를 코드 버그로 단정하기 전에 **하니스 한정 원인**을 한 겹 의심한다(쿠키 Secure 평문 미전송,
  테스트 순서가 pause 시점을 지나침). memory probe-deeper의 검증판.
