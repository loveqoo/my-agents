# 034 — 공유·실 DB 검증은 불변식 + 델타로 (절대수치 금지)

## 문제
실행 중인 실 DB(기존 데이터 다수)를 상대로 API를 검증할 때, "빈 DB" 가정으로
절대수치(`counts.live == 37`)를 단언하면 깨진다. 깨끗한 픽스처 DB를 매번 띄우는
건 무겁고, 운영 유사 환경 검증의 가치를 잃는다.

## 해법 두 축
1. **불변식(invariant)** — 데이터 내용과 무관하게 항상 참인 성질만 단언.
   - 합/일관성: `total(status=X) == counts[X]`, 페이지 순회 수집수 == total.
   - 구조: 비중복(distinct), 정렬 단조(started_at desc), 셰이프 `{items,total,counts}`.
   - 경계: offset>total → `items=[]`·total 유지, limit 클램프 → 422, 미지 status → 폴백.
2. **델타(delta)** — 고유 prefix로 N건 주입 → 주입 전후 차이만 단언 → 삭제로 자가정리.
   - `counts.all` 증가분 == 주입 건수, `counts.live` 증가분 == 주입 중 live 분포.
   - prefix(`sess_v034_`)로 자기 행만 식별·삭제 → 실데이터 불간섭, 재실행 안전.

## 왜 우월한가
절대수치 단언은 "내 환경 = 깨끗"이라는 숨은 가정을 품어, 환경이 다르면 거짓 실패/
거짓 통과를 낸다. 불변식·델타는 그 가정을 제거 → 어떤 선행 데이터에서도 같은 신호.

## 적용 신호
- 검증 대상이 공유/영속 DB이고, 매 실행 클린 픽스처가 비쌀 때.
- "이 값이 정확히 N이어야 한다"를 쓰고 싶을 때 → "이 값들의 *관계*가 성립해야 한다"로
  바꿀 수 있는지 먼저 본다.
