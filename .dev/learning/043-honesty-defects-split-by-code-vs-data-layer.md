# 043 — "표시가 거짓"인 결함은 코드층·데이터층으로 갈라서 고친다

## 상황

어드민 승인 배지가 "2"인데 목록은 8개였다(#12). 한 증상처럼 보이지만 *원인이 두 층*에 있었다:

- **코드 거짓**: 배지가 mock 상수(`PENDING_APPROVALS=2`)를 읽고 있었다(하드코딩).
- **데이터 거짓**: DB에 시드된 가짜 승인 행 8개가 pending으로 남아 목록을 부풀렸다.

## 교훈

**"보여주는 값이 실제와 다르다"류 결함은 거의 항상 *코드가 잘못 읽는 것*과 *데이터 자체가 가짜인
것*이 겹쳐 있다. 한 배치에 욱여넣지 말고 고치는 층으로 갈라라.**

- 코드 거짓 → 순수 코드 diff. 리뷰·롤백 쉽고, 비파괴적이라 바로 출하 가능.
- 데이터 거짓 → 삭제/마이그레이션. 비가역이라 dry-run·적대 리뷰가 필요(→ memory
  adversarial-review-before-destructive-ship). 출하 리듬이 완전히 다르다.

섞으면 "코드 고쳤는데 화면은 여전히 이상"(데이터가 안 고쳐짐)하거나, 코드 PR에 DELETE가 섞여
롤백이 위험해진다. 045(배지 코드 정직화)와 050(가짜 시드 제거 dry-run)을 분리하니 045는 깨끗한
코드 단위로, 050은 데이터 정리로 각자의 검증 강도를 가졌다.

## 적용 트리거

화면 수치/상태가 실제와 어긋날 때. "이 거짓은 *코드가 잘못 읽어서*인가, *데이터가 가짜라서*인가,
둘 다인가?"를 먼저 묻고 배치를 그 축으로 가른다. 관련: [[038-adversarial-review-finds-what-invariants-miss]].
