# 099 — 확장점을 "런타임"에 두느냐 "저작시점"에 두느냐가 신뢰경계를 가른다

> 스펙 099, retrospect 080. 참고: spec 085(신뢰 레지스트리 §보안경계 — eval/import 경로 없음)·
> 089(isinstance fail-closed 게이트)·[[installed-guard-isnt-covering-guard]]·[[why-build-multi-agent-platform]].

## 배운 것

### 1. "어떻게 확장하나"가 아니라 "언제 확장하나"가 보안 속성을 결정한다

같은 기능("사용자가 새 에이전트 플로우를 만들어 재사용")이라도 **확장점을 어디에 두느냐**가 신뢰경계를
가른다. 시각 그래프 빌더는 런타임에 *사용자 산출물(그래프 JSON)을 로드*해야 하고, 그러면 결국
**문자열→클래스 해석**(어떤 노드 구현을 쓸지 이름으로 찾기) 경로가 생겨 "임의 문자열 eval/import
없음"(085 U5) 불변식과 충돌한다. 반면 **저작시점 코드젠**(스킬이 모듈을 파일로 쓰고 레지스트리에
정적 import 2줄 추가)은 산출물이 **커밋·리뷰·재기동**을 거치므로 신뢰경계가 그대로 닫힌다 — dict
조회만 하는 신뢰 레지스트리는 손도 안 댄다.

핵심: 확장 기능을 설계할 때 "런타임에 사용자 입력을 코드로 해석하는 경로가 생기는가"를 먼저 물어라.
생기면 그 기능은 신뢰경계를 위협한다. **확장점을 저작시점으로 밀면**(codegen + 정적 등록 + 재기동)
같은 사용자 가치를 신규 보안 표면 거의 0으로 얻는다. "저마찰 생성"은 런타임 동적 로딩이 아니라
*저작 자동화(스킬)*로도 달성된다.

### 2. 폐기가 옳은 답일 때가 있다 — 대체 수단이 기존 시임에 얹히면

무거운 기능(시각 빌더)을 **폐기**하고 목적만 남겨 **이미 검증된 시임**(Protocol+신뢰 레지스트리+
conformance 검증) 위에 저작 워크플로우로 재구현하면, 신규 런타임 표면과 검증 부담이 급감한다.
수용 게이트도 **새로 발명하지 말고** 기존 검증(089 conformance·085 구조/실 노드 타임라인)을 재사용해
"생성 → 자가 수용" 루프를 닫는다. 데모 산출물 1종(route)이 스펙의 자체 실증이 된다.

### 3. 생성 코드의 검증가능성은 "순수함수 분리"로 산다

코드젠 산출물에서 **분기/판정 로직은 노드 클로저 밖 모듈 수준 순수함수**로 빼라(`classify_route(text)
->"a"/"b"`). 그래야 단위 테스트가 실 LLM 없이 결정성을 단언한다. 클로저 안에 묻으면 모델이 필요한
통합에서만 확인 가능 → 검증 사다리의 값싼 단(단위)이 비어버린다. 스킬 규약에 명문화할 것.

## 훅

- 확장 기능 설계 시 첫 질문: "런타임에 사용자 입력을 코드로 해석하는 경로가 생기나?" 생기면 신뢰경계
  위협 — 확장점을 **저작시점(codegen+정적 등록+재기동)**으로 밀어 신규 보안 표면을 0으로.
- 무거운 UI 기능은 폐기가 답일 수 있다 — 목적을 **기존 검증된 시임 + 저작 스킬**로 재구현하면 표면 급감.
  수용 게이트는 발명 말고 기존 conformance/구조 검증 재사용(생성→자가수용 루프).
- 코드젠 산출물의 분기·판정 로직은 **모듈 순수함수**로 분리 → 단위에서 모델 없이 결정성 단언.
- codex에 넘길 땐 `.claude/skills/` 등 스코프 금지 경로를 리뷰 번들에서 빼고 **코드 파일만**(문서는
  보안 검토 대상 아님, 손실 0).
