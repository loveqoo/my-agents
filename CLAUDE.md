# 작업 방식 (Operating Manual)

이 저장소의 모든 작업은 아래 **6단계 루프**를 반복한다. 작업 성격에 따라
특정 단계가 불필요할 수 있다 — 이 경우 **사람과 합의 + 사유 명시** 후 스킵한다.

> **항상 현재 단계를 명시한다.** 매 응답 머리에 `[단계 N — 이름]` 형태로
> 지금 6단계 루프 중 어디에 있는지 표기하여, 사람과 AI가 루프 위치를
> 늘 공유한다.

---

## 6단계 루프

### 1. Scaffolding — 구조 점검·정비
- 작업에 맞는 구조인지 먼저 확인한다. 변경이 필요할 때만 구조를 손댄다.
- 그 외에는 **역할별 폴더링**을 유지한다.
- `docs/`는 **참고 전용**. 특별한 명령이 없으면 수정하지 않는다. (인간이 작성: 아키텍처·컨벤션·스펙)
- `tests/`에는 검증에 필요한 모든 파일을 둔다.
- `.dev/`에는 작업의 흔적을 기록한다 — 학습(learning), 문제 해결 과정(troubleshooting), 작업 로그·디버깅 히스토리·실험 결과·스크래치 패드.

### 2. Context — 계층적·점진적 기록
- **스코프를 활용**한다: User 영역 → Project 영역 → Folder 영역. **하위가 상위를 덮어쓴다**(위로 갈수록 추상적).
- 한꺼번에 다 적지 않는다. "이런 상황에서는 이걸 참고해"라고 안내하여 **필요한 것만 읽게** 한다.
- 폴더별 맥락은 해당 폴더의 컨텍스트 파일(예: 하위 `CLAUDE.md` 또는 README)에 둔다.

### 3. Planning — 계획 → 검토 → 승인 → 실행
- AI가 계획을 작성하고, 사람이 검토·피드백, 수정 후 **승인**받는다. 그 다음 실행한다.
- 무엇을 할지 모호하면 **의도가 잡힐 때까지 질문**하여 요구사항/태스크를 도출한다.
- 작성된 플랜은 반드시 **파일로 남긴다** → `docs/spec/NNN-{name}.md`.

### 4. Execution — 플랜 실행
- 중요한 기능은 **테스트로 동작을 먼저 확인**한다.
- 실행 방법을 영리하게 고민한다. 독립 작업은 **병렬 실행**을 적극 고려한다.
- 명확한 완료 기준이 있으면 **Ralph Loop**(완료 조건까지 반복)도 고려한다.

### 5. Verification — 결과 검증
- 해당 작업의 **완료 조건을 상기**한 뒤 검증한다.
- **자가검증은 최대한 피한다.** 가능하면 서브에이전트나 타사 모델(codex, copilot 등)로 **비판적 검증**을 받는다.
- 화면 검증이 필요하면 브라우저 제어/MCP/스크린샷 등 다양한 수단을 고민한다.
- 사이드이펙트가 예상되면 **dry-run**을 적극 고려한다.
- 사람 의존 검증은 지양한다. **반복적이고 빠른 검증**이 우선이다.

### 6. Compounding — 회고·자산화
- 무엇이 잘못됐고 무엇을 배웠는지 기록한다 → `.dev/learning/`, `.dev/troubleshooting/`.
- **AI 자신의 회고는 `.dev/retrospect/NNN-{name}.md`** (AI 영역). 인간이 작성하는 회고만 `docs/retrospect/`(인간 영역, 명령 필요).
- 같은 작업을 반복하면 **스킬/커맨드**로 도출한다.
- 같은 실수를 반복하면 Scaffolding에서 **훅이나 장치**로 막는다.
- 이 과정을 반복하며 절차를 선명하게, 실수를 적게, 사용자 의도에 가깝게 만든다.

## 폴더 구조

```
.claude/                        클로드 설정 파일
.dev/                           (AI 영역 — 작업의 흔적)
  learning/                     배운 것
  troubleshooting/              문제 해결 과정
  retrospect/  NNN-{name}.md     AI 회고
docs/                           (인간 영역, 참고 전용 — 명령 없이 수정 금지)
  spec/        NNN-{name}.md     스펙·플랜
  adr/         NNN-{name}.md     아키텍처 결정 기록
  retrospect/  NNN-{name}.md     인간 회고
tests/                          검증 자산
CLAUDE.md                       루트 컨텍스트(본 문서)
```

문서 파일은 `001-`처럼 **3자리 일련번호 + 케밥케이스** 이름을 쓴다.

---

## 빠른 합의 규칙

- 매 응답 머리에 **현재 단계 `[단계 N — 이름]` 표기**.
- `docs/`는 **명령 없이 수정 금지**.
- 모든 플랜은 **파일로** 남긴다.
- 단계 스킵은 **합의 + 사유** 필요.
- 검증은 **타자(서브에이전트·타사 모델)** 우선, 자가검증 지양.
