# 043 — 컨텍스트 경량화: 자산 인덱스 층 + Context 규칙 교정 + 완료 스펙 아카이브

> 워크플로 정비(Scaffolding 발). 코어 기능 변경 아님 — **Context 단계의 읽기 비용을 줄여
> compaction 주기를 늦추는** 구조 개선. 합의: L1+L2+L3 (2026-06-27).
> 관련: CLAUDE.md(6단계 루프·Context 규칙), learning 023(규칙은 보이는 강제 필요),
> memory dont-skip-context-recall-learnings(recall이 복리 동력).

## 배경·문제

자산이 누적되며 **Context 단계의 읽기량이 compaction을 당기고** 있다(사용자 보고).

| 영역 | 개수 | 용량 |
|---|---|---|
| `docs/spec/` | 42 | 372KB / 4140줄 |
| `.dev/learning/` | 41 | 168KB / 1292줄 |
| `.dev/retrospect/` | 32 | 168KB / 1428줄 |
| 합계 | 115 | ~708KB |

**근본 원인은 CLAUDE.md Context 규칙 자체.** 현재 규칙은 "작업 키워드로 grep해 관련 자산을
**읽고** 적용점을 적는다" — grep 매칭 파일을 **통째로 읽게** 돼 있다. 자산 115개에서 키워드
하나에 5~10개가 매칭되고 각 100~180줄이라, Context 한 번에 수백~수천 줄을 흡수한다.

**핵심 관찰:** 메모리(`MEMORY.md`)는 이미 "한 줄 인덱스 → 필요한 1개만 full" 구조로 가볍게 돈다.
learning/retrospect/spec에는 그 인덱스 층이 없어 매번 full을 grep·통독한다. **같은 패턴을
자산에도 입히는 것**이 가장 큰 레버다.

## 비범위

- learning 물리 통합(L4): 원문 손실·재작성 리스크라 본 스펙에서 **제외**. 인덱스에서 "관련 묶음"
  표시로 대체. 필요해지면 후속.
- 자산 자동 인덱스 갱신 훅(스캐폴딩 장치): 본 스펙은 **수동 갱신 규칙**까지. 훅 자동화는 후속(이번
  L1~L3가 자리잡고 갱신 누락이 실제로 관찰되면).
- 코드/런타임 변경 없음. 순수 문서·규칙 정비.

## 목표(완료 조건, 측정 가능)

1. **인덱스 3종 신설**: `docs/spec/INDEX.md`, `.dev/learning/INDEX.md`, `.dev/retrospect/INDEX.md`.
   각 항목당 **정확히 한 줄** = `- NNN 제목 — 후크 [키워드,키워드]`. 모든 기존 항목 누락 없이 포함
   (spec 42 + learning 41 + retrospect 32 = 115줄 + 헤더).
2. **CLAUDE.md Context 규칙 교정**: "grep해 full 읽기" → "**INDEX 먼저 읽고, 한 줄로 불충분할
   때만 그 항목 full 열기**". `참고 자산:` 줄 형식은 유지(적용점 명시 의무 유지).
3. **완료 로드맵 스펙 아카이브**: 스펙 034~042(P0~P5, 100% 완료)를 `docs/spec/archive/`로 이동.
   `INDEX.md`에는 `[archived]` 표시로 남겨 참조는 가능하되 기본 grep 핫패스에서 제외.
4. **자기검증 가능**: 인덱스 줄 수 == 실제 파일 수(아카이브 포함), 모든 NNN이 인덱스에 1회씩 존재,
   깨진 참조 없음 — 스크립트로 확인.

## 설계

### L1. 인덱스 포맷 — `INDEX.md` (3종)

각 디렉터리 루트에 한 파일. 라인 포맷(메모리 `MEMORY.md` 미러):

```
# Spec Index  (docs/spec/)

- 001 시스템 개요 — 전체 아키텍처·용어 정리 [overview,architecture,terms]
- 034 세션 페이징 — offset/limit+total 서버 페이징·status 필터 [session,paging,offset-limit] [archived]
- 042 A2A 실호출 — 외부 에이전트 JSON-RPC 런타임 호출+SSRF [a2a,external,ssrf,jsonrpc] [archived]
...
```

- **후크**: 그 자산을 "언제 열어야 하는지" 한 줄. 키워드는 grep 친화적(영문 소문자).
- 첫 줄(`# NNN — 제목`)이 전 파일에 일관 → 인덱스 생성이 거의 기계적(제목 추출 + 후크·키워드 부여).
- `[archived]` 꼬리표 = `docs/spec/archive/`로 이동된 항목(L3).

### L2. CLAUDE.md Context 규칙 교정

§2(Context) 의 `참고 자산:` 문단을 다음 취지로 교정:

> **`참고 자산:` 줄 필수** — 비자명한 작업은 시작 시 **먼저 해당 `INDEX.md`(spec/learning/retrospect)를
> 읽고**, 작업 키워드로 관련 항목을 **인덱스에서 고른다**. 한 줄 후크로 충분하면 그 줄을, 불충분하면
> **그때만 해당 full 파일을 연다**. Context 응답에 `참고 자산: [번호-제목 → 적용점]`을 적는다.
> 인덱스에 없는 새 grep이 필요하면 grep도 병행하되, **full 통독은 후크가 불충분한 항목에 한정**한다.

§"빠른 합의 규칙"의 Context 줄도 동일 취지로 갱신. (learning 023: 규칙은 보이는 강제가 있어야
지켜진다 → "INDEX를 먼저 읽는다"를 명시적 첫 동작으로 박아 강제력 부여.)

### L3. 완료 스펙 아카이브 — `docs/spec/archive/`

- `docs/spec/archive/` 신설, 스펙 034~042(9개) 이동(`git mv`).
- `INDEX.md`는 이동된 항목을 `[archived]`로 표기하되 경로는 `archive/NNN-...`로 갱신.
- **주의**: 스펙 간 상호 참조(`docs/spec/NNN`) 링크가 깨지지 않게 확인. 아카이브 대상이 참조하는/
  참조되는 링크를 grep해 경로 갱신 또는 인덱스 경유로 정리.
- 아카이브 기준 = **로드맵 완료분(034~042)만**. 001~033은 토대·현황 정의가 많아 핫패스 유지.

### 갱신 규칙(수동, 본 스펙 범위)

- 새 spec/learning/retrospect를 만들 때 **같은 턴에 해당 INDEX.md에 한 줄 추가**(Compounding 단계
  체크리스트에 편입). 누락 시 다음 Context에서 인덱스-파일 수 불일치로 드러남(목표 4의 자가검증).
