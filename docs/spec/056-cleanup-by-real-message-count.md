# 056 — 세션 정리: 카운터를 진실로 유지하고 거짓말의 출처(seed)를 고친다

> **반전 주의**: 초안 제목은 "실제 메시지 수 기준으로(카운터 불신)"이었으나, codex 적대리뷰가
> 그 접근의 실결함(윈도우 모드 고턴 세션 오삭제)을 짚어 **카운터 유지 + seed 진실화**로 반전했다.
> 아래 설계는 최종(출하)본이며, 폐기된 메시지-수 접근과 반전 사유를 §반전에 남긴다.

## 배경 / 문제
세션 정리 배치(스펙 049)의 턴 절은 `Session.turns < min_session_turns`로 판정한다.
그런데 `turns`는 **비정규화 카운터**다 — `_persist`가 턴마다 +1 하지만, 실제 `Message` 행과
독립적으로 저장된다. `seed.py`(118~124줄)가 데모 세션을 **메시지 0행인데 turns=6/14/21로
부풀려** 심는 탓에, *실제로는 빈 세션*이 *고턴*으로 보여 저턴 정리에서 **명시적으로 보존**된다.

실측(2026-06-28): `session-cleanup`이 오늘 실제로 돌아 118건을 지웠는데도(min_session_turns=4),
시드 세션 sess-8f21(turns=6)·sess-7a05(turns=14)·sess-4b10(turns=21)은 전부 real_msgs=0인데
카운터가 ≥4라 살아남았다. 사용자 보고("저턴이면 지우기로 했는데 왜 남아있나")의 근본 원인.
→ learning 058(비정규화 카운터 ≠ 실제 행 수)이 실제 정리 워크플로를 문 사례.

## 결정(최종)
턴 절은 **카운터 `Session.turns`를 진실로 유지**한다. 부풀린 거짓말의 **유일한 출처였던 seed를
0으로 고친다**. 카운터가 정확하면 시드 빈 세션(turns=0)은 정리되고, 윈도우 모드 고턴 세션은
보존된다 — 신호를 바꾸지 않고 거짓말의 출처를 고치는 게 옳다(learning 059).

(초안 결정은 "카운터 불신·실제 Message 행으로 판정"이었다. codex 적대리뷰가 이를 뒤집었다 → §반전.)

## 설계

### A. 배치 — 턴 절은 카운터 유지, 주석으로 의도 명문화 (`batch/jobs.py:cleanup_sessions`)
- 로직은 **원본 그대로**: `(Session.turns < min_turns) & (Session.last_activity < idle_cutoff)`.
- 왜 카운터인가를 주석으로 못박는다: `turns`는 메시지 행의 *캐시가 아니라 더 완전한 진실*이다.
  `_persist`는 턴마다 +1 하는데 `persistHistory=false`(윈도우 모드)면 메시지를 일부러 안 남기므로
  **`turns ≥ 메시지행수`가 항상 성립**. 메시지 행 수로 세면 100턴 윈도우 세션(메시지 0행)을
  저턴으로 오인해 지운다. 카운터를 신호로 두고 거짓말의 출처(seed)를 고치는 게 옳다.
- 나이 절·IDLE_GUARD·HIL pending 제외·idempotent·CASCADE 전부 **무변경**.

### B. 시드 — turns 진실화 (`seed.py`)
- 메시지 0행인 데모 세션의 `turns`(와 동반 lie인 `tokens`)를 **0으로** 맞춘다. 카운터가
  실제 활동과 일치(파생값을 거짓으로 두지 않음). 데모 세션은 status·channel로 다양성을
  유지하되, 빈 세션은 정직하게 0턴으로 보이고 정리 대상이 된다(사용자 의도: "빈 세션 정상 정리").
- (비범위: 데모에 실제 메시지를 심어 히스토리가 보이게 하는 것은 AskUserQuestion의 옵션3 — 추후.)

## 반전 — 왜 메시지-수 접근을 폐기했나 (codex 적대리뷰)
초안은 learning 058("비정규화 카운터 ≠ 실제 행 수")을 따라 턴 절을 `count(user Message) < N`으로
바꿨고, 양방향 드리프트 적대 테스트도 통과했다(거짓 green). codex가 짚은 실결함:
- `persistHistory=false`(윈도우 모드, `schemas.py`)는 메시지를 **의도적으로 미저장**(`chat.py:330`)
  하면서 `turns`는 계속 +1(`chat.py:335`). → 100턴 윈도우 세션도 메시지 0행이라 `user_turns<N`에
  걸려 삭제. 세션 메타·연속성까지 "저턴 정크"로 소실되는 **의미 보존 실패**.
- 통찰: 여기서 `turns`는 행의 *캐시*가 아니라 *더 완전한 진실*이다(`turns ≥ 메시지행수`). 058의
  "실제 행을 세라"는 *카운터가 행의 캐시일 때만* 옳다. 카운터가 행으로 안 남는 사건까지 합법적으로
  세면, 행-세기는 정보를 잃는다. 잘못된 카운터 값의 수리는 *기록자(seed)를 고치는 것*이지 신호를
  바꾸는 게 아니다. → learning 059.

## 리스크 / 주의
- **파괴성**: 정리는 dry-run으로 먼저 대상 확인 후 실삭제. 실 DB 적용·정리 활성화는 사용자 명시 시에만.
- 현재 라이브 설정은 `min_session_turns=None`·`session_cleanup_cron=None`(정리 비활성·자동 미발화).
  시드 빈 세션을 실제로 비우려면 ① 라이브 행 turns→0 갱신(거짓 라벨 제거) ② 정리 활성화 또는 직접 삭제.
