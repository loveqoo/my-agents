# 044 — UI 테스트 피드백 정리 마스터 플랜

> Scaffolding 발. 2026-06-28 UI 테스트 후 사용자가 제기한 14건을 메뉴별로 트리아지하고
> 의존순으로 배열한 **지도**. 각 배치는 자체 스펙(045~050)으로 실행·검증·커밋(per-spec).
> 관련: CLAUDE.md(6단계 루프), 스펙 033(로드맵 — 같은 "전수조사 후 의존순 배열" 패턴),
> memory adversarial-review-before-destructive-ship(파괴적 정리), learning 025(시드-레지스트리 drift).

## 배경

하루치 자율 구현(로드맵 P0~P5 완료) 후 사용자가 어드민 UI를 실제로 돌려 보고 14건의
피드백을 냈다. 성격이 섞여 있다 — 직접 질문, 버그, 설계 변경, 누락 작업, 기능 신규,
재료 정리, 파괴적 데이터 정리. 한 스펙에 안 들어가므로 메뉴별로 가른다.

## 조사로 답한 직접 질문 (서브에이전트 코드 전수조사)

- **#3 코드 에이전트 별도 등록 버튼의 의도**: `Agent.source`가 3값(`ui`|`code`|`external`,
  models.py:203)이고 각 생명주기가 다르다 — `ui`=콘솔 편집·버전관리, `code`=레포 배포·읽기전용·
  엔드포인트 재동기화, `external`=A2A 카드 소유·로컬설정 없음. 버튼 3개는 이 3소스를 반영. **의도 있음.**
- **#6 127.0.0.1 vs localhost**: 같은 호스트 분할이 아니라 **다른 두 서비스** — `MLX (local)`=
  `localhost:8045/v1`(실 로컬 MLX), `Mock LLM`=`127.0.0.1:8000/_remote/v1`(라이브 없이 결정적
  테스트용 self-call mock, 스펙 024). 임의 분할 아님. → **라벨만 명확히** 다듬는다(047에 편입).
- **#2 A2A 등록 시 실존 테스트**: 추측과 달리 **이미 검증함** — agents.py:341이 저장 전 Agent Card
  fetch+필수필드+SSRF 가드(agent_card.py). 단 (a) 시드가 가짜 `Acme Translate (A2A)`를 검증 없이
  주입, (b) 카드 fetch 성공 ≠ 실행 엔드포인트 liveness. → **liveness ping 보강**(045).

## 의존·순서 근거

- **045 먼저**: 가장 작고 명확한 버그(#12). 빠른 승리 + 신뢰 회복.
- **046 권한/MCP 정리**: 047 통합 전에 재료를 비워야 통합 UI가 깔끔. 독립적이라 병행 가능.
- **047**은 가장 큼(스키마·UI·외부 카탈로그) — 046 후.
- **048 RAG**, **049 세션**은 047과 독립 → 사이사이 진행 가능.
- **050 파괴적 정리는 맨 뒤**: 위 기능 변경이 시드/스키마를 건드리므로 데이터 정리는 그 후에
  해야 "정리했더니 기능 작업이 다시 더럽힘" 회귀를 피한다. memory 규칙대로 dry-run+적대 검증.
- **테스트 위생(#13 후단)**: 045부터 모든 작업에서 **테스트가 만든 유저·세션·A2A는 그 작업
  끝에 정리**(또는 정리 스크립트에 누적). 새로 안 더럽힌다.

## 검증(공통 원칙)

- 각 배치 스펙은 **완료 조건을 측정 가능하게** 두고, **타자(서브에이전트/codex) 적대 검증** 우선.
- 파괴적 경로(050)는 **dry-run + 적대 리뷰** 필수(memory adversarial-review-before-destructive-ship).
- UI 변경은 **브라우저로 직접 확인**(Playwright+시스템 Chrome, memory verify-ui-in-browser-proactively).
