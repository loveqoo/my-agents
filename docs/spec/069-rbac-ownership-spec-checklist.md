# 069 — RBAC/소유권 경계 스펙 체크리스트 (회고를 재발방지 훅으로)

## 배경 / 문제

RBAC·멀티테넌트 작업 066→067→068이 연속으로 **같은 부류의 결함**을 사후 봉합했다.
읽기 경로만 스코핑하고 *다른 입구*를 빠뜨리는 패턴이다.

- **067**: `sessions.py` list/item 읽기 스코핑. 적대 codex가 "chat resume가 우회한다"를 짚음(범위 밖→068).
- **068 D1/D3 (learning 069)**: 읽기를 막아도 **쓰기/resume 입구**가 소유자를 *덮어쓰면* 탈취·존재 오라클.
- **068 D6 (learning 070)**: 그 입구를 조이고 나니 **생성 입구**(`_create_approval` lazy-create)가 소유자를
  *안 박던* 게 드러남 → 정당한 본인조차 못 잇는 **자가-잠금**. happy-path 라이브 12/12 초록인데 적대자가 발견.

retrospect 055가 자인한다: *"데이터를 생성·변경하는 경로의 소유권을 같은 표에 넣었다면 P1을 사전에 짚었을 것."*
즉 교훈(learning 069·070)은 남았으나, 다음 스펙 설계 때 **상기를 강제하는 구조 훅이 없다**. 회고는
다음 작업 Context에서 상기·적용될 때만 복리가 된다 — 이 스펙은 그 상기를 *강제*하는 장치다.

## 목표 (사용자 결정)

RBAC/멀티테넌트/소유권 경계 스펙을 쓸 때 **반드시 채워야 할 체크리스트**를 만들고, 스펙 작업 시
**자동 로드되는 위치**(폴더 컨텍스트 `docs/spec/CLAUDE.md`)에 박아 상기를 강제한다.

## 비목표

- 코드 변경 없음(068까지 실코드는 이미 봉인). 이 스펙은 *프로세스* 자산이다.
- 모든 스펙에 강제하지 않는다 — **소유권/테넌시 경계가 걸린 스펙**에 한정. 단 그 판정은 작성자
  *자가판정*이 아니라 **객관 신호**로 가른다(`user_id`·테넌트 컬럼 접촉, `_own_scope`/`_visible_or_404`/
  `_assert_*owns` 헬퍼가 닿는 코드 변경) — 자가판정에 맡기면 "경계인 줄 몰라서 빠뜨린" 066의 맹점이
  트리거 단계로 옮겨갈 뿐(installed-guard-isnt-covering-guard).

## 체크리스트 본문 (= 폴더 CLAUDE.md에 박을 내용)

소유권/테넌시 경계가 걸린 스펙이면 설계에서 아래를 **명시적으로 답**한다.

1. **입구 열거(닫힌 집합으로)** — 이 자원을 만지는 *모든* 입구를 센다: `list` / `item(/{id})` /
   `create` / `update` / **`delete`(비가역이라 update와 별도 호명 — `memory_routes.py`·`agents.py`에
   소유권 걸린 삭제 입구 실재)** / `resume`·재진입 / **lazy-create(부수 생성)** / **외부 프로토콜 입구**
   (A2A `contextId` 멀티턴 등 — 내부 HTTP 라우트만 세면 샌다). "주 영속 경로 하나"만 보면 샌다(learning 070).
   *이 목록 자체를 닫힌 집합으로 취급*하지 말고, 새 입구 종류가 나오면 여기에 추가한다.
2. **각 입구의 소유권 처리** — 입구마다: (a) 읽기는 owner 스코프를 **SELECT WHERE에** 밀어넣어
   거부행을 *로드조차 안 함*(타이밍 오라클까지 차단). **fetch-then-check(먼저 로드 후 사후 거부)는
   그 자체가 잔존 타이밍 오라클**이다(`sessions.py` detail/messages/end = retro 056 [P3-1]) — 신규는
   SELECT-WHERE로, 기존 fetch-then-check는 "미봉합 잔존"으로 스펙에 *명시 기록*해 가시화한다,
   (b) 쓰기/resume은 소유자 **덮어쓰기 금지** 불변식, (c) 생성/lazy-create은 **생성 시 1회** 소유자
   스탬프(learning 069·070), (d) **비-SQL 저장소**(mem0/pgvector처럼 전역 id·벡터DB·외부 KV로
   SELECT-WHERE를 못 받는 경우): 멤버십을 list로 확인 후 *별도 호출*로 지우면 **check-then-act
   TOCTOU 창**이 열린다(`memory_routes.py` `_assert_user_owns`→`delete_memory`) → 삭제/수정 *호출
   자체에 owner를 파라미터로 묶거나* 판정→실행을 원자로 만든다.
3. **단일 헬퍼** — 소유권 판정·이전을 *한 헬퍼*로 통일(068 `_own_scope`/`_next_owner`)해 드리프트 0.
4. **존재 비노출** — 볼 수 없는 행은 **404로 접어** 403(존재하나 금지)과 구분하지 않음
   = 열거 오라클 제거(learning 068).
5. **검증 사다리 3런(비겹침)** — ① 단위 시맨틱, ② **실 인프라 통합(seed+restart)**: casbin `g`·시드
   drift·요청간 글루는 이 런만 잡음, ③ **적대 타자(codex)**: "보장 목록의 *여집합*"(다른 입구·다른
   방향)을 던짐. happy-path 초록은 *상상한 실패*만 본다.
6. **자가-잠금 별도 핀** — 게이트를 조였으면 "조임이 *정당한* 접근을 막진 않나"를 따로 테스트
   (learning 070). 조임은 탈취만이 아니라 자기 데이터 차단으로도 샌다.

## 적대 검증 결과 (서브에이전트, 반영 완료)

체크리스트 작성 직후 적대 타자가 "066~068 결함을 *사전에* 잡았겠는가 + 놓친 부류"를 검증.
과거 4개 결함(읽기만 스코핑·resume 탈취·lazy-create 자가잠금·404오라클)은 전부 매핑 통과.
**미래 부류 P1 갭 2개를 발견해 반영**: (1) 입구 열거에 `delete` 누락(실코드 `memory_routes.py:170`·
`agents.py:613`), (2) 비-SQL 저장소(mem0 전역 id)의 check-then-act **TOCTOU** 미커버. 추가로 A2A 외부
입구·fetch-then-check 잔존(P2)·트리거 자가판정(P3)을 항목 1·2(a)(d)·트리거에 흡수. 이 발견 자체가
체크리스트 항목 5③(적대 런)의 자기적용 — "입구 열거가 닫힌 집합이 아님"을 적대자가 짚어 보강했다.
